# mylogin CI/CD 部署手册 · 操作步骤(具体命令 + 验证)

> 本文是"逐步操作 + 检查":每一步敲什么、配什么、怎么确认成功。
> 整体架构与分阶段逻辑见配套文档 **《部署方案-总览.md》**。章节与总览的"阶段"一一对应。
>
> 全文凡 `<HOST_A_IP>` `<HOST_B_IP>` 处,都替换成你的真实公网 IP。

---

## 准备清单(动手前先备齐)

- [ ] 主机 A 公网 IP:`<HOST_A_IP>`
- [ ] 主机 B 公网 IP:`<HOST_B_IP>`
- [ ] 两台主机的 root 或 sudo 账号(用于一次性准备)
- [ ] 两个 GitHub 仓库(后端、前端各一个,代码已能 push)
- [ ] 本机装了 `ssh` / `ssh-keygen`(Windows 可用 Git Bash 或 PowerShell 自带 OpenSSH)

---

## 第一部分:代码改动(对应总览·阶段一)

> 改完后本地照样能跑;先在本地确认能编译/构建,再进入第二部分。

### 1.1 后端:外置 DB 密码和 JWT secret

**文件 `src/main/resources/application.yml`**:

```yaml
spring:
  datasource:
    url: jdbc:sqlserver://106.55.7.17:1433;databaseName=mylogin;encrypt=true;trustServerCertificate=true
    username: sa
    password: ${DB_PASSWORD}          # ← 原明文删掉,改环境变量
    driver-class-name: com.microsoft.sqlserver.jdbc.SQLServerDriver
  # ...其余不变...

jwt:
  secret: ${JWT_SECRET}               # ← 同样改环境变量
  access-expiration-minutes: 1
  refresh-expiration-days: 7

# 新增:CORS 允许的源,默认本地 dev,生产由环境变量覆盖
cors:
  allowed-origin: ${CORS_ALLOWED_ORIGIN:https://localhost:5173}
```

> 改后**本地 IDE 运行也要提供环境变量**:IDEA Run Configuration → Environment variables 加 `DB_PASSWORD=...;JWT_SECRET=...`。仓库里就不再有明文。

### 1.2 后端:CORS 源改成可配置

**文件 `src/main/java/com/t/d/mylogin/config/SecurityConfig.java`**:

```java
// 新增字段(配合 import org.springframework.beans.factory.annotation.Value)
@Value("${cors.allowed-origin}")
private String allowedOrigin;

// 把 corsConfigurationSource() 里:
//   config.setAllowedOrigins(List.of("https://localhost:5173"));
// 改成:
config.setAllowedOrigins(List.of(allowedOrigin));
```

### 1.3 前端:接口走 nginx 同源

**文件 `src/api/index.js`**:

```js
const api = axios.create({
  baseURL: '/',           // ← 原为 'https://localhost:8888';改走同源相对路径,nginx 反代到后端
  withCredentials: true,  // 保持不变
})
```

> 改后前端发 `/api/auth/login` 这类相对路径,落到主机 A,nginx 转给主机 B。本地调试可临时切回 `https://localhost:8888`,上线用 `/`。

### 1.4 前端:vite 构建时不读本地证书(否则 CI 必崩)

**文件 `vite.config.js`**(现无条件读 `./certs/*.pem`,CI 上没证书会构建失败):

```js
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import * as fs from 'node:fs'

export default defineConfig(({ command }) => ({
  plugins: [vue()],
  // 仅本地 dev(serve)启用 HTTPS 自签证书;build(CI)不读证书,避免流水线失败
  ...(command === 'serve' && {
    server: {
      port: 5173,
      https: {
        cert: fs.readFileSync('./certs/localhost.pem'),
        key: fs.readFileSync('./certs/localhost-key.pem'),
      },
    },
  }),
}))
```

**✅ 本部分验收**:本地 `mvn clean package`(带环境变量)与 `npm run build` 都成功。

## 第二部分:服务器一次性准备(对应总览·阶段二)

### 2.0 生成专用部署密钥(在你本机执行)

```bash
ssh-keygen -t ed25519 -C "github-deploy" -f ./mylogin_deploy -N ""
cat ./mylogin_deploy        # 私钥 → 稍后进 GitHub Secrets
cat ./mylogin_deploy.pub    # 公钥 → 稍后放两台主机
```

---

### 2.1 主机 B 准备(后端)

SSH 登录主机 B(sudo 账号),依次执行:

**(1) 安装 JDK 17**

```bash
sudo yum install -y java-17-openjdk-headless
java -version          # 确认 17.x
```
> 若源里没有 java-17(老 CentOS 7):导入 Adoptium 源后 `sudo yum install -y temurin-17-jdk`。

**(2) 建目录和运行用户**

```bash
sudo useradd -r -s /sbin/nologin mylogin
sudo mkdir -p /opt/mylogin/incoming
sudo mkdir -p /etc/mylogin
```

**(3) 写运行时密钥文件**

```bash
sudo tee /etc/mylogin/backend.env >/dev/null <<'EOF'
DB_PASSWORD=MyDev@2024#Sql
JWT_SECRET=bXlsb2dpbi1qd3Qtc2VjcmV0LWtleS0yMDI0LWRlbW8=
CORS_ALLOWED_ORIGIN=https://<HOST_A_IP>
EOF
sudo chmod 600 /etc/mylogin/backend.env
```
> 替换 `<HOST_A_IP>`;生产请换强随机 JWT_SECRET。

**(4) 创建 systemd 服务**

```bash
sudo tee /etc/systemd/system/mylogin-back.service >/dev/null <<'EOF'
[Unit]
Description=mylogin backend
After=network.target

[Service]
User=mylogin
WorkingDirectory=/opt/mylogin
EnvironmentFile=/etc/mylogin/backend.env
ExecStart=/usr/bin/java -jar /opt/mylogin/app.jar
SuccessExitStatus=143
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable mylogin-back            # 开机自启(暂无 jar,先不 start)
```

**(5) 创建部署用户 deploy + 装公钥**

```bash
sudo useradd -m deploy
sudo mkdir -p /home/deploy/.ssh
echo "ssh-ed25519 AAAA....github-deploy" | sudo tee /home/deploy/.ssh/authorized_keys
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
```

**(6) 授权 deploy 写 jar 目录 + 重启服务**

```bash
sudo chown -R deploy:deploy /opt/mylogin
echo 'deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart mylogin-back, /usr/bin/systemctl status mylogin-back' | sudo tee /etc/sudoers.d/deploy
sudo chmod 440 /etc/sudoers.d/deploy
```

**(7) 防火墙:8888 只放行主机 A**

```bash
sudo firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="<HOST_A_IP>/32" port port="8888" protocol="tcp" accept'
sudo firewall-cmd --reload
```
> 22 保持公网可达(GitHub runner 出口 IP 不固定),靠密钥认证。**云厂商安全组**同样放行 22(全网)、8888(仅主机 A)。

---

### 2.2 主机 A 准备(nginx)

SSH 登录主机 A:

**(1) 安装 nginx**

```bash
sudo yum install -y nginx
```

**(2) 生成自签证书**

```bash
sudo mkdir -p /etc/nginx/certs
sudo openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout /etc/nginx/certs/mylogin.key \
  -out    /etc/nginx/certs/mylogin.crt \
  -subj "/CN=<HOST_A_IP>"
```

**(3) 建前端站点目录**

```bash
sudo mkdir -p /var/www/mylogin
```

**(4) 写 nginx 站点配置**

```bash
sudo tee /etc/nginx/conf.d/mylogin.conf >/dev/null <<'EOF'
server {
    listen 80;
    server_name <HOST_A_IP>;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name <HOST_A_IP>;

    ssl_certificate     /etc/nginx/certs/mylogin.crt;
    ssl_certificate_key /etc/nginx/certs/mylogin.key;

    root  /var/www/mylogin;
    index index.html;

    # SPA(vue-router history 模式):找不到的路径回退 index.html,否则深链接刷新 404
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端反代:/api/* → 主机B 8888(自签 HTTPS,故 verify off)
    location /api/ {
        proxy_pass https://<HOST_B_IP>:8888;
        proxy_ssl_verify off;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF
```
> 默认 `/etc/nginx/nginx.conf` 里若自带监听 80 的 `server {}` 与之冲突,注释掉它。

**(5) SELinux 放行(CentOS 必做)**

```bash
sudo setsebool -P httpd_can_network_connect 1          # 允许 nginx 反代发起连接
sudo yum install -y policycoreutils-python-utils
sudo semanage fcontext -a -t httpd_sys_content_t "/var/www/mylogin(/.*)?"
sudo restorecon -Rv /var/www/mylogin
```

**(6) 创建 deploy 用户 + 公钥 + 授权写站点**

```bash
sudo useradd -m deploy
sudo mkdir -p /home/deploy/.ssh
echo "ssh-ed25519 AAAA....github-deploy" | sudo tee /home/deploy/.ssh/authorized_keys
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chown -R deploy:deploy /var/www/mylogin
```

**(7) 启动 nginx + 放行 80/443**

```bash
sudo systemctl enable --now nginx
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --permanent --add-service=https
sudo firewall-cmd --reload
```
> 云厂商安全组同样放行 80、443。

**✅ 本部分验收**:两台主机各有可被部署私钥登录的 `deploy` 用户;主机 B 的 systemd 服务已 enable(待 jar);主机 A 的 nginx 已运行。

## 第三部分:GitHub 配置(对应总览·阶段三)

### 3.1 配置 Secrets(两个仓库都配)

仓库 → **Settings → Secrets and variables → Actions → New repository secret**:

| Secret 名 | 值 |
|-----------|-----|
| `HOST_A_IP` | 主机 A 公网 IP |
| `HOST_B_IP` | 主机 B 公网 IP |
| `DEPLOY_USER` | `deploy` |
| `DEPLOY_SSH_KEY` | `mylogin_deploy` **私钥**全文(含 `-----BEGIN...END-----`) |

> DB 密码、JWT secret **不进 GitHub**,已在主机 B 的 `/etc/mylogin/backend.env`。

### 3.2 后端仓库工作流

**后端仓库**新建 `.github/workflows/deploy.yml`:

```yaml
name: CI-CD Backend
on:
  push:
    branches: [ main ]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '17'
          cache: maven

      # 先测后打包:测试失败这步即红,部署不执行(测试门禁)
      - name: Build & Test
        run: mvn -B clean package

      - name: Upload jar to Host B
        uses: appleboy/scp-action@v0.1.7
        with:
          host: ${{ secrets.HOST_B_IP }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          source: "target/*.jar"
          target: "/opt/mylogin/incoming"
          strip_components: 1

      - name: Activate & Restart
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.HOST_B_IP }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            set -e
            cp /opt/mylogin/app.jar /opt/mylogin/app.jar.bak 2>/dev/null || true
            mv /opt/mylogin/incoming/*.jar /opt/mylogin/app.jar
            sudo systemctl restart mylogin-back
            sleep 3
            sudo systemctl --no-pager status mylogin-back
```

### 3.3 前端仓库工作流

**前端仓库**新建 `.github/workflows/deploy.yml`:

```yaml
name: CI-CD Frontend
on:
  push:
    branches: [ main ]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm

      - name: Install & Build
        run: |
          npm ci
          npm run build

      - name: Clean remote web root
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.HOST_A_IP }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: rm -rf /var/www/mylogin/*

      - name: Upload dist to Host A
        uses: appleboy/scp-action@v0.1.7
        with:
          host: ${{ secrets.HOST_A_IP }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          source: "dist/*"
          target: "/var/www/mylogin"
          strip_components: 1
```
> `appleboy/*-action` 版本号若失效,去其 GitHub 页面换最新 tag。

## 第四部分:首次跑通与验证(对应总览·阶段四)

> 先后端、再前端,逐个验证。

### 4.1 后端首次部署

1. 提交第一部分的后端改动,push 到 `main`。
2. 仓库 **Actions** 看 `CI-CD Backend`:`Build & Test` 绿 = 单测过、jar 打好;`Activate & Restart` 末尾打印 `active (running)`。
3. 主机 B 自查:

```bash
sudo systemctl status mylogin-back               # active (running)
sudo journalctl -u mylogin-back -n 50            # 看启动日志,确认连上 DB
curl -k https://localhost:8888/api/auth/login -X POST \
     -H 'Content-Type: application/json' -d '{}'  # 通了会返回 401/400(而非连接拒绝)
```

### 4.2 前端首次部署

1. 提交第一部分的前端改动,push 到 `main`。
2. Actions 看 `CI-CD Frontend` 跑绿。
3. 浏览器开 `https://<HOST_A_IP>`:
   - 自签证书安全警告 → "继续前往"(正常现象);
   - 登录页 → 用 DB 里的账号登录;
   - F12 → Network,确认请求打到 `https://<HOST_A_IP>/api/auth/login` 且 200;
   - 登录后刷新页面仍保持登录。

### 4.3 端到端成功标志

- 登录成功、能看用户列表;
- 刷新页面不掉登录;
- 多开标签页,一个登出另一个跟着登出;
- 全程 Network 无 CORS 报错。

## 第五部分:日常使用与排错(对应总览·阶段五)

### 5.1 日常发版

```bash
git push origin main
```
其余 test → build → deploy 全自动。Actions 红了点进去看哪步失败。

### 5.2 回滚后端到上一版本

CI 自动保留上一版为 `app.jar.bak`。SSH 主机 B:

```bash
sudo -u deploy bash -c 'cp /opt/mylogin/app.jar.bak /opt/mylogin/app.jar'
sudo systemctl restart mylogin-back
```

### 5.3 故障速查

| 现象 | 多半原因 | 处理 |
|------|----------|------|
| `/api` 返回 **502** | nginx 连不上主机 B | 查主机 B 服务是否 running、防火墙/安全组是否放行 8888 给主机 A、SELinux 是否做了 `httpd_can_network_connect` |
| 首页 **403** | SELinux 没给 /var/www 上下文 | 重跑 2.2 (5) 的 `semanage`+`restorecon` |
| 深链接刷新 **404** | nginx 少 SPA 回退 | 确认 `try_files ... /index.html` 在位 |
| 部署步骤 **Permission denied** | 公钥没装对 / deploy 无权限 | 核对两台主机 `authorized_keys`、`/opt/mylogin` 与 `/var/www/mylogin` 属主是 deploy |
| 后端起不来报 **password/secret 为空** | env 文件没配/路径错 | 查 `/etc/mylogin/backend.env`、systemd 的 `EnvironmentFile` |
| `npm run build` 在 CI **找不到 certs** | 没做 1.4 改动 | 按 1.4 改 vite.config.js |
