# 從傳統模板頁面到 SPA：架構演進全景

---

## 第一部分：傳統模板頁面（MPA）的本質與前世今生

### 本質抽象

> **「瀏覽器是一個啞終端，服務器是唯一的智慧中心。」**

所有的計算、所有的邏輯、所有的狀態，全部在服務端完成。瀏覽器只負責一件事——把服務端送來的 HTML 畫出來。它沒有自己的判斷，沒有自己的狀態，就像一台只會顯示畫面的顯示器。

### 誕生背景

互聯網誕生於 1990 年代初，設計目標非常單純：**在科研機構之間共享文檔**。Tim Berners-Lee 發明 HTTP 和 HTML 的初衷，是讓物理學家能互相閱讀對方的論文。

這個場景決定了一切設計選擇：服務器是圖書館，瀏覽器是讀者，HTTP 是借書協議。請求-響應，一問一答，天然合理。

### 歷史脈絡

**第一階段：純靜態文檔時代（1990s 初）**

服務器上就是一堆 `.html` 文件，用戶請求什麼就返回什麼。沒有「後端邏輯」的概念，服務器就是一個文件服務器。

**第二階段：動態內容的需求出現（1990s 中）**

商業互聯網興起，網站需要顯示動態內容——今天的新聞、用戶的訂單、數據庫裡的商品。靜態 HTML 滿足不了需求，**CGI（通用網關接口）** 出現——服務器收到請求後調用腳本程序，腳本生成 HTML 字符串返回給瀏覽器。這是「服務端渲染」的雛形。

**第三階段：模板引擎時代（1990s 末 ～ 2000s）**

CGI 太原始，拼 HTML 字符串難寫難維護。各種**模板引擎**誕生：PHP（1994）、JSP（1999）、ASP（1996）。思路一致：以 HTML 為骨架，嵌入佔位符，服務端渲染時把真實數據填進去。

這是 MPA 的黃金時代。服務端控制一切：路由、鑒權、數據查詢、頁面生成。瀏覽器收到的永遠是「已填好數據的完整 HTML」。

**第四階段：AJAX 出現，裂縫開始（2000s 中）**

2005 年 Google 發布 Gmail 和 Google Maps——頁面不刷新，數據卻在實時更新。背後是 **AJAX**：瀏覽器可以在後台悄悄發 HTTP 請求，只更新頁面的一部分。

瀏覽器第一次有了**主動獲取數據**的能力，不再完全被動。但整體架構還是 MPA，AJAX 只是補丁。

**第五階段：MPA 的極限（2000s 末）**

隨著 Web 應用越來越複雜，傳統 MPA 的矛盾積累到臨界點：

- 每次跳轉整頁刷新，用戶體驗割裂
- 服務端同時承擔業務邏輯和頁面渲染，職責混雜難以擴展
- 前後端代碼耦合嚴重，Java 程序員要寫 JSP
- 移動端崛起，同一套業務邏輯要服務 Web 和 App，服務端返回 HTML 對 App 毫無意義

這些矛盾催生了新的架構。

### 一句話總結

> MPA 是「文檔共享」思想的延伸：服務器是權威的內容提供者，瀏覽器是忠實的內容展示者。它在簡單場景下完美，在複雜應用場景下顯露出「服務端承擔太多、瀏覽器承擔太少」的結構性矛盾。

## 第二部分：SPA 的本質與前世今生

### 本質抽象

> **「瀏覽器是一台完整的應用運行機器，服務器退化為純粹的數據提供者。」**

職責發生了根本性的翻轉。服務端不再管「頁面長什麼樣」，只管「數據是什麼」。瀏覽器不再只是顯示器，它變成了一個完整的應用容器——路由、渲染、狀態管理、交互邏輯，全部在瀏覽器裡運行。

### 誕生背景：三股力量同時推動

**第一股：用戶體驗的壓力。**
Google Maps 和 Gmail 重新定義了用戶對 Web 的期待——頁面應該像桌面應用一樣流暢。整頁刷新的交互模式變得不可接受。

**第二股：移動端的崛起。**
iPhone 在 2007 年發布，移動 App 爆發。同一套業務邏輯要同時服務 Web、iOS、Android。服務端返回 HTML 對 App 沒有意義——App 需要純粹的數據（JSON），自己決定怎麼渲染。後端 API 化成為必然。

**第三股：JavaScript 的成熟。**
V8 引擎（2008）讓 JavaScript 執行速度提升數十倍。瀏覽器第一次有了真正承擔複雜邏輯的運算能力。

### 歷史脈絡

**第一階段：先驅探索（2000s 中）**

開發者用 jQuery 手動管理 DOM、手動發 AJAX 請求、手動更新頁面片段。沒有框架，全靠自己搭架子。代碼極難維護，但方向明確：**瀏覽器可以做更多事。**

**第二階段：框架誕生，SPA 正式成形（2010 ～ 2013）**

- **Backbone.js（2010）**：第一個主流前端 MVC 框架，把路由、模型、視圖的概念帶入瀏覽器
- **AngularJS（2010，Google）**：雙向數據綁定讓 UI 和數據自動同步
- **React（2013，Facebook）**：組件化和虛擬 DOM，UI 是「狀態的函數」

SPA 核心思想確立：**一次加載，客戶端路由，數據驅動視圖。**

**第三階段：工程化爆發（2014 ～ 2018）**

SPA 帶來了新問題：應用越來越大，JavaScript 代碼量激增。這推動了前端工程化全面爆發：模塊化規範、Webpack 打包工具、npm 生態、Vue（2014）成為主流。

前端從「寫幾個腳本」演變為完整的工程學科，前後端徹底分離成為行業標準。

**第四階段：SPA 的反思與修正（2019 至今）**

SPA 暴露出兩個嚴重問題：

- **首屏性能**：要先下載整個 JS bundle 才能渲染，用戶先看到白屏
- **SEO 問題**：搜索引擎拿到的是空白 HTML 殼，無法索引內容

這催生了 **SSR（服務端渲染）的回歸**——Next.js（React）、Nuxt.js（Vue）應運而生。首屏由服務端渲染好 HTML 直接返回，後續交互接管為 SPA 模式運行。

這是架構的螺旋式上升——不是退回 JSP 時代，而是在更高層次整合兩個時代的優點。

### SPA 的代價

每一次架構演進都有代價，SPA 把複雜性從服務端轉移到了前端：

- 服務端不再管路由，前端要自己實現路由系統
- 服務端不再管鑒權攔截，前端要自己寫路由守衛
- 服務端不再維護頁面狀態，前端要自己做狀態管理
- 瀏覽器刷新會清空一切，前端要自己處理狀態恢復

### 一句話總結

> SPA 是移動互聯網時代「前後端職責重新分工」的產物：後端專注數據與業務規則，前端承擔應用運行環境的全部責任。它解決了 MPA 的體驗問題，代價是把原本隱藏在服務端的複雜度暴露給了前端工程師。

## 第三部分：職責遷移——服務端與前端的重新分工

### 基準視角

傳統 MPA 時代，服務端承擔兩類職責：

- **業務職責**：處理業務規則、操作數據庫、保障數據安全
- **呈現職責**：決定頁面長什麼樣、跳到哪裡、狀態是什麼

SPA 的本質，就是把**呈現職責整體遷移給前端**，**業務職責繼續留在服務端**。

### 已遷移到前端的職責

| 職責 | 傳統 MPA（服務端做） | SPA（前端做） |
|---|---|---|
| 頁面路由 | 服務端 URL 映射到 Controller，返回對應 HTML | 前端 Router 攔截 URL，切換對應組件 |
| 頁面渲染 | 服務端模板引擎填充數據，生成完整 HTML | 前端框架根據數據動態渲染組件 |
| 跳轉控制 | 服務端 redirect/forward 決定下一個頁面 | 前端 router.push() 決定切換哪個組件 |
| 鑒權攔截（呈現層） | Interceptor 攔截未登錄請求，重定向登錄頁 | 路由守衛攔截未登錄導航，跳轉登錄頁 |
| 頁面狀態維護 | Session 存當前用戶，每次請求服務端自動讀取 | Pinia 在瀏覽器內存維護當前用戶狀態 |
| 狀態恢復 | 天然不需要，Session 服務端持久 | 刷新後主動調接口，從服務端重新同步 |
| 錯誤頁面 | 服務端返回 404.html、500.html | 前端路由配置 fallback 組件 |
| 加載反饋 | 不需要（整頁刷新本身就是等待） | 前端自己管理 loading 狀態 |

### 保留在服務端的職責

| 職責 | 為什麼不能遷移 |
|---|---|
| 業務規則校驗 | 前端校驗可被繞過，業務規則必須在可信環境執行 |
| 數據庫讀寫 | 數據庫不能暴露給瀏覽器 |
| 鑒權校驗（數據層） | 每個 API 必須在服務端驗證身份，前端守衛只是體驗優化 |
| 密碼加密與比對 | 加密算法和密鑰不能在瀏覽器裡暴露 |
| Session 存儲 | 服務端持有的認證憑據，是信任鏈的根基 |
| 數據權限過濾 | 哪些數據用戶能看，必須服務端過濾後再返回 |
| 事務管理 | 數據一致性只能在服務端保障 |
| 第三方服務調用 | API Key 不能暴露在前端代碼裡 |

### 容易混淆的關鍵點：前後端各做一遍，但目的不同

遷移後，很多職責變成了前後端各做一遍，層次和目的不同：

| 職責 | 前端做的目的 | 後端做的目的 |
|---|---|---|
| 鑒權攔截 | 體驗——未登錄時不讓進入受保護頁面，避免白屏或報錯 | 安全——防止未授權請求拿到真實數據 |
| 表單校驗 | 體驗——即時反饋，減少無效請求 | 安全——防止非法數據進入系統 |
| 錯誤處理 | 體驗——友好提示用戶 | 安全——不暴露系統內部細節 |

> **前端的攔截可以被繞過，後端的攔截不能被繞過。** 前端做的那一層永遠只是體驗優化，不是安全保障。

## 附：傳統 MPA 各職責對應的 Java 類

### 路由分發

```
DispatcherServlet          ← 所有請求的總入口
HandlerMapping             ← 解析 URL，找到對應 Controller 方法
ViewResolver               ← 把視圖名稱解析為實際模板文件
ThymeleafViewResolver      ← 渲染模板，整頁返回給瀏覽器
```

Controller 返回視圖名稱（非 JSON），由 ViewResolver 渲染成完整 HTML 頁面。SPA 之後，Controller 只返回 JSON，頁面由前端 Vue Router 決定顯示哪個組件。

### 鑒權攔截

**Servlet Filter 層（最外層）：**
```
javax.servlet.Filter
    └── OncePerRequestFilter      ← Spring 封裝，保證每次請求只執行一次
            └── Spring Security 過濾器鏈
```

**Spring MVC Interceptor 層：**
```
HandlerInterceptor
    └── preHandle()               ← 進 Controller 前攔截，等價於前端 beforeEach
    └── postHandle()              ← Controller 執行後攔截
    └── afterCompletion()         ← 視圖渲染完後執行
```

前端的 `router.beforeEach` 本質是在瀏覽器裡複製了一個 `HandlerInterceptor.preHandle()`，只是攔截的是組件切換，而不是 HTTP 請求。

### 頁面狀態維護

```
HttpSession                ← 核心，服務端內存的 key-value 存儲
@SessionAttributes         ← 把 Model 數據自動同步進 Session
SessionScope               ← Spring Bean 的 Session 作用域
```

傳統模式下前端沒有「狀態管理」概念。SPA 之後，HttpSession 還在，但前端多了 Pinia 作為鏡像副本。

### 狀態恢復

傳統 MPA 根本不存在「狀態恢復」問題——Session 存在服務端，JSESSIONID Cookie 自動攜帶，服務端永遠知道用戶是誰，瀏覽器刷不刷新無所謂。

```
HttpSessionSecurityContextRepository   ← 把認證信息存進 Session
SessionManagementFilter                ← 管理 Session 的創建與同步
```

SPA 之後，HttpSession 依然工作。新增問題是：Pinia 刷新後丟失，需要主動調後端接口重新同步狀態。

### Pinia 與四大域的對比

Pinia 在四大域裡沒有精確對應物：

| 四大域 | 生命週期 | Pinia 對比 |
|---|---|---|
| PageContext | 單次頁面渲染，響應後銷毀 | ❌ 太短，Pinia 跨頁面存活 |
| Request | 單次 HTTP 請求 | ❌ 太短 |
| Session | 瀏覽器會話（服務端保存） | ⚠️ 最接近，但數據在瀏覽器內存而非服務端 |
| Application | 服務器整個生命週期 | ❌ 太長 |

> **Pinia 最接近的比喻**：Java 進程堆內存中的靜態變量。靜態變量在 JVM 運行期間一直存在，JVM 重啟則清空。Pinia 在瀏覽器 JS 引擎運行期間一直存在，頁面刷新則清空。

## 宏觀結論

SPA 的職責遷移本質上是一次**關注點分離**：

> 服務端專注於「數據是否正確、操作是否合法」——它是系統的**信任邊界**。  
> 前端專注於「界面是否流暢、體驗是否友好」——它是系統的**呈現邊界**。  
> 兩者之間只剩下一條乾淨的契約：**API 接口**。  
> 這條契約讓同一個後端可以同時服務 Web、iOS、Android，也讓前後端可以獨立演進。
