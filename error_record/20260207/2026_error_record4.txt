  Keycloak 加密流程：
    1. 随机生成一个 AES-256 会话密钥
    2. 用 AES-256-GCM 加密 Assertion 明文内容
    3. 用 mylogin-sp.crt（SP 公钥）+ RSA-OAEP 加密这个 AES 密钥
    4. 把加密后的 AES 密钥和密文一起发给 Spring Boot

  Spring Boot 解密流程：
    1. 用 mylogin-sp-private.pem（SP 私钥）解密出 AES 密钥
    2. 用 AES 密钥解密出明文 Assertion

 一、原始"改 URL 必发请求"是怎么工作的

  浏览器内部有两个独立的东西，原来强制绑定在一起:

  ① Session History Stack   ← 浏览器维护的历史记录栈（前进/后退用）
  ② Navigate 算法           ← 发 HTTP 请求、替换页面的执行流程

  原来改 URL 的任何方式都同时触发两者:

  window.location.href = '/dashboard'
  // ① 往历史栈压入 /dashboard
  // ② 同时触发 navigate 算法 → HTTP GET /dashboard → 整页替换
  // 两件事永远绑在一起，不可分割

● 一、基准线：原始行为（两个机制强制绑定）

  用户点 <a href="/dashboard">
           │
           ▼
  ① Session History Stack:  压入 '/dashboard'        ✓
  ② Navigate 算法:          完整执行                  ✓
                              → HTTP GET /dashboard
                              → 收到响应
                              → 整页替换

  两件事永远同时发生，没有办法只要①不要②。

  ---
  二、Hash 路由：进入了 Navigate，但走了捷径分支

  window.location.hash = '#/dashboard'
           │
           ▼
  ② Navigate 算法:  启动了                            ✓
           │
           ├─ 判断：path 部分变了吗？/app → /app
           │   → 没变，只有 # 后面变了
           │
           ▼
           走"同文档导航"分支:
             HTTP 请求                                ✗ 跳过
             整页替换                                 ✗ 跳过
             触发 hashchange 事件                     ✓
           │
           ▼
  ① Session History Stack:  压入新条目               ✓

  Hash 路由没有绕过 Navigate 算法，它是进去了，但走了一条 if 分支提前返回。 不发请求不是因为绕开了 Navigate，而是 Navigate 内部有一个判断："如果只改了 fragment，就不用去网络取新文档"。

  ---
  三、History API：完全绕过 Navigate 算法

  history.pushState(null, '', '/dashboard')
           │
           ▼
  ② Navigate 算法:  根本没有被调用                   ✗
           │
           ▼
  直接操作:
  ① Session History Stack:  压入 '/dashboard'        ✓
    地址栏更新显示           ✓
    HTTP 请求                                        ✗（Navigate 没跑，自然没有）
    整页替换                                         ✗

  pushState 在引擎层是一条直达历史栈的通道，完全绕开了 Navigate 算法。

  四、用代码结构来理解（Java 风格）

  // Navigate 算法的伪代码骨架
  void navigate(String url) {
      historyStack.push(url);              // 第一步：压栈

      if (onlyFragmentChanged(url)) {      // Hash 路由走这里
          fire("hashchange");
          return;                          // ← 提前返回，不发请求

      String html = httpClient.get(url);   // 第二步：发 HTTP 请求
      replacePage(html);                   // 第三步：替换页面
  }

  // pushState 完全不调用上面这个方法
  void pushState(String url) {
      historyStack.push(url);              // 直接操作栈
      addressBar.update(url);              // 更新地址栏显示
      // navigate() ← 根本没有调用


  Hash 路由 = navigate() 被调了，但 return 提前出来了。
  History API = navigate() 根本没被调。

  五、一张对照表钉死

  ┌───────────────────┬────────────┬────────────────┬────────────┬──────┐
  │       操作        │ ① 压历史栈 │ ② Navigate算法 │ 发HTTP请求 │ 换页 │
  ├───────────────────┼────────────┼────────────────┼────────────┼──────┤
  │ <a> 点击          │ ✓          │ 完整执行       │ ✓          │ ✓    │
  ├───────────────────┼────────────┼────────────────┼────────────┼──────┤
  │ location.href = x │ ✓          │ 完整执行       │ ✓          │ ✓    │
  ├───────────────────┼────────────┼────────────────┼────────────┼──────┤
  │ #fragment 变化    │ ✓          │ 进入走快捷分支 │ ✗          │ ✗    │
  ├───────────────────┼────────────┼────────────────┼────────────┼──────┤
  │ pushState()       │ ✓          │ 完全绕过       │ ✗          │ ✗    │
  └───────────────────┴────────────┴────────────────┴────────────┴──────┘

  三种情况①永远发生，区别只在②怎么处理。

  ---
  一句话钉死: Hash 路由和 History API 都保留了①（历史栈），区别是对②的处理方式不同——Hash 是进入 Navigate 算法后提前 return（利用 fragment-only 的特殊分支），History API 是连 Navigate
  算法都不调用（直接操作历史栈）。两种不同的"让②不完整执行"的路径。
