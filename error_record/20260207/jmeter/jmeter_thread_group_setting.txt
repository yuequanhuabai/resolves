 为什么 Thread Group 要这样设置

  核心公式：TPS = 并发线程数 / 平均响应时间(秒)

  已知条件：
    目标 TPS = 0.9
    假设平均响应时间 ≈ 2~3 秒（一次审批提交涉及 DB + Flowable）

  反推线程数：
    线程数 = TPS × 平均响应时间
           = 0.9 × 3
           = 2.7 → 向上取整，至少 3 个线程

  设 5 个线程的原因：留余量，因为响应时间会波动
    如果某一刻响应变慢到 5 秒：
      3 个线程最大 TPS = 3 / 5 = 0.6 → 达不到 0.9
      5 个线程最大 TPS = 5 / 5 = 1.0 → 仍然能满足 0.9

  每个参数的作用

  Number of Threads = 5
    含义：模拟 5 个并发用户

  Ramp-Up Period = 10
    含义：10 秒内逐步启动 5 个线程（每 2 秒启动 1 个）
    为什么不设 0（全部同时启动）？
      避免瞬间并发冲击服务器，更接近真实场景

    0s    2s    4s    6s    8s    10s
    |     |     |     |     |     |
    线程1 线程2 线程3 线程4 线程5    全部就绪

  Loop Count = Infinite
    含义：每个线程无限循环发请求，直到 Duration 时间到
    为什么不用固定次数（如 108）？
      固定次数的问题：
        如果响应快（1秒），108次 × 5线程 = 540次，可能 2 分钟就跑完
        如果响应慢（5秒），可能 10 分钟跑不完 108 次就超时了
      用 Infinite：不管响应快慢，都跑满 10 分钟，由 Timer 控制节奏

  Duration = 600
    含义：线程组最多运行 600 秒（10 分钟），时间到自动停止所有线程

  它们之间的配合关系

            Thread Group                    Constant Throughput Timer
       ┌──────────────────────┐          ┌──────────────────────────┐
       │ 提供 5 个"工人"      │          │ 控制"工人"的工作节奏     │
       │ 持续工作 10 分钟     │    +     │ 整体保持 54 次/分钟     │
       │ 每人不限循环次数     │          │ 即 0.9 次/秒            │
       └──────────────────────┘          └──────────────────────────┘

  没有 Timer：5 个线程全力发请求，TPS 可能飙到 2~5，远超 0.9
  没有足够线程：响应变慢时，Timer 想控制 0.9 TPS 也做不到（线程不够用）

  两者配合：Timer 在每次请求前插入等待时间，让 5 个线程"慢慢做"

  ---
  Constant Throughput Timer 参数详解

  Target throughput = 54.0

  单位是 samples/minute（每分钟请求数），不是每秒！

  0.9 TPS = 0.9 × 60 = 54 次/分钟

  Calculate Throughput based on — 5 个选项详解

  假设场景：5 个线程，目标 54 次/分钟

  ---
  1. This thread only

  每个线程独立计算，每个线程都尝试达到 54 次/分钟

  线程1: 54 次/分钟
  线程2: 54 次/分钟
  线程3: 54 次/分钟
  线程4: 54 次/分钟
  线程5: 54 次/分钟
  ──────────────────
  总计:  270 次/分钟 = 4.5 TPS ← 远超目标！

  ❌ 不适合你的场景
  如果用这个选项，Target 要设 54/5 = 10.8

  2. All active threads

  所有线程组的所有线程共同分担 54 次/分钟

  如果你有多个线程组（buylist + benchmark + ...），
  所有线程组的活跃线程加在一起分担这 54 次

  线程组A的3个线程 + 线程组B的5个线程 = 8个线程共享 54次/分钟

  ⚠️ 跨线程组共享，单交易压测时不建议用

  3. All active threads in current thread group ✅ 推荐

  当前线程组内的所有线程共同分担 54 次/分钟

  buylist 线程组的 5 个线程共同分担：
    线程1: ~10.8 次/分钟
    线程2: ~10.8 次/分钟
    线程3: ~10.8 次/分钟
    线程4: ~10.8 次/分钟
    线程5: ~10.8 次/分钟
    ──────────────────────
    总计:  54 次/分钟 = 0.9 TPS ✅

  ✅ 单交易压测最合适的选项
  只看自己线程组，不受其他线程组影响

  4. All active threads (shared)

  和选项 2 类似，所有线程组的线程共同分担 54 次/分钟

  区别在于 "shared"：
    非 shared：每个线程独立计算延迟时间，可能有短时波动
    shared：   所有线程共享一个全局计数器，分配更均匀

  实际差异很小，主要在高 TPS 场景下才有明显区别

  5. All active threads in current thread group (shared)

  和选项 3 类似，当前线程组内共同分担 54 次/分钟

  区别同上：shared 版本用全局计数器，分配更均匀

  适用于对 TPS 精度要求更高的场景

  总结对比

  选项                                          │ 范围       │ 总 TPS
  ──────────────────────────────────────────────┼────────────┼──────────
  1. This thread only                           │ 每个线程   │ 54 × 5 = 270/min
  2. All active threads                         │ 所有线程组 │ 54/min
  3. All active threads in current thread group │ 当前线程组 │ 54/min ✅
  4. All active threads (shared)                │ 所有线程组 │ 54/min（更均匀）
  5. All active threads in current group(shared)│ 当前线程组 │ 54/min（更均匀）

  单交易负载测试 → 选 3 或 5
  混合交易负载测试（多线程组同时跑）→ 选 2 或 4
