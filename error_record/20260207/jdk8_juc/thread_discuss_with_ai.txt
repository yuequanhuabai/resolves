● 这两个类是一个用 sun.misc.Unsafe 手动实现
  CAS（Compare-And-Swap）计数器的示例。

  本质逻辑

  核心问题： 多线程环境下 i++ 不是原子操作（读-改-写三步），会导致竞态条件。如何
  在不加锁的情况下实现线程安全的自增？

  解决方案：CAS 无锁自旋

  CAS 的语义是："我认为当前值是 old，如果确实是，就更新为 
  next；如果不是（被别的线程改了），就什么都不做，返回 
  false"。配合死循环不断重试，直到成功为止——这就是自旋（spin）。

  这正是 AtomicInteger 的底层原理，这里用 Unsafe 手动复现了它。

  逐类分析

  MyCASCounter — CAS 计数器

  1. volatile int value（第9行）—
  保证多线程间的可见性，每次读取都从主存获取最新值。
  2. 静态初始化块（第15-28行）— 两件事：
    - 通过反射获取 Unsafe 单例（因为其构造函数是 private 的）
    - 用 objectFieldOffset 获取 value 字段在对象内存中的偏移量，CAS
  操作需要它来定位要修改的内存地址
  3. incrementAndGet()（第30-38行）— 核心方法：
  for (;;) {                    // 死循环（自旋）
      int old = value;          // 读当前值
      int next = old + 1;       // 计算新值
      if (UNSAFE.compareAndSwapInt(this, VALUE_OFFSET, old, next))
          return next;          // CAS 成功就返回，失败则重试
  }
  3. compareAndSwapInt 是一条 CPU 原子指令（x86 上对应
  CMPXCHG），硬件级别保证原子性。

  MyCASCounterTest — 测试类

  - 10 个线程，每个线程自增 1000 次
  - join() 等待所有线程执行完毕
  - 预期输出：result: 10000（因为 CAS 保证了线程安全）

● 有问题，顺序反了。

  你的理解（错误）

  ▎ 先改值为 next → 再检查 old 有没有被别人改过 → 如果被改了就回滚

  这是"先写后验"，如果真这样做，在写和验之间别的线程也能看到这个中间状态，就不是
  原子操作了。

  正确的语义

  先比较，再交换（Compare-And-Swap，名字本身就说明了顺序）：

  compareAndSwapInt(this, VALUE_OFFSET, old, next)

  这一条指令做的事情是：

  1. 读取 this 对象在 VALUE_OFFSET 偏移处的当前内存值
  2. 比较：当前内存值 == old 吗？
  3. 如果相等 → 说明没有别的线程改过，把内存值写为 next，返回 true
  4. 如果不等 → 说明已经被别的线程改了，什么都不做，返回 false

  关键点：步骤 1-3 是 CPU 级别的一条原子指令（x86 的
  CMPXCHG），中间不可能被其他线程插入。不存在"先改了再检查再回滚"的过程。

  用生活类比

  你的理解相当于：先把钱花了，再看看钱包余额对不对，不对就退款。

  正确的逻辑是：先看钱包余额对不对，对了才花钱——而且看和花是同一个不可分割的动作
  。

● 大方向对了，但有两个细节需要纠正。

  第一个问题："等待下次获取 CPU 时间片"——不准确

  看回代码：

  for (;;) {
      int old = value;       // 1. 读当前值
      int next = old + 1;    // 2. 算新值
      if (UNSAFE.compareAndSwapInt(this, VALUE_OFFSET, old, next)) {
          return next;       // 3. CAS 成功，退出
      }
      // CAS 失败，不会挂起等待，而是立刻回到循环顶部重新读值、重试
  }

  CAS 失败后，线程不会主动让出 CPU，而是立刻在当前时间片内重试——重新读
  value、重新算 next、重新
  CAS。这就是"自旋"的含义：原地转圈重试，不休眠、不阻塞。

  这也是 CAS 的优缺点所在：
  - 优点：不用进内核做线程切换，竞争小的时候非常快
  - 缺点：如果竞争激烈，大量线程空转浪费 CPU

  第二个问题：对 old 值的理解——基本正确

  你说的核心流程是对的，更准确地表述：

  1. int old = value — 线程读取当前值，存到自己的局部变量 old
  2. 在执行 CAS 的那一瞬间，CPU 原子地比较内存中的实际值和 old
  3. 相等 → 没被别人改过 → 写入 next，返回 true
  4. 不等 → 被别人改了 → 什么都不做，返回 false，立刻重试

  所以关键区别就是：失败后是立刻自旋重试，不是被动等待下次调度。

> 还有一个问题，如果当前线程让出时间片之后，它是什么线程状态呢？它又是存储在哪  
里了？获取下次cpu时间片的方式又是什么？                                         

● 这个问题需要分清 Java 线程状态和 OS 层面的调度两个层次。

  一、先明确场景

  在 CAS 自旋循环中，线程不会主动让出时间片。但时间片用完后，OS
  会强制抢占（preempt），把 CPU 分给别的线程。

  二、线程状态

  从 Java 的角度（Thread.State），这个线程始终是 RUNNABLE。

  RUNNABLE ——  表示"可运行"，包含两种 OS 层面的子状态：
    ├── Running   —— 正在占用 CPU 执行
    └── Ready     —— 没有 CPU，在就绪队列里排队等调度

  关键点：时间片用完被抢占后，Java 状态不变，仍然是 RUNNABLE。它不是
  WAITING，也不是
  BLOCKED，因为它没有主动等待任何东西——它随时可以继续跑，只是暂时没拿到 CPU
  而已。

  只有以下操作才会离开 RUNNABLE：
  - synchronized 竞争锁失败 → BLOCKED
  - wait() / park() / sleep() → WAITING / TIMED_WAITING

  CAS 自旋不涉及这些，所以状态不变。

  三、被抢占时存储在哪里

  OS 会做一次上下文切换（Context 
  Switch），把线程的执行现场保存到内核中的数据结构：

  保存的内容（线程上下文）：
    ├── 程序计数器（PC）—— 执行到哪条指令了
    ├── CPU 寄存器值   —— 包括通用寄存器、栈指针等
    └── 栈指针         —— 指向该线程的线程栈

  这些信息保存在 OS 内核为每个线程维护的 TCB（Thread Control Block）
  中。线程栈本身一直在内存里，不需要额外保存。

  四、如何再次获取 CPU

  OS 调度器（Linux 默认是 CFS）从**就绪队列（Ready Queue）**中选下一个线程：

  就绪队列：[线程A, 线程B, 线程C, ...]
                  ↑
              调度器按优先级/公平性策略选一个
                  ↓
           恢复它的上下文 → 继续执行

  线程不需要做任何事，OS 会自动把它重新调度上 CPU，恢复之前保存的寄存器和
  PC，从被打断的那条指令继续往下跑。

> 就绪队列是什么，在java中有对应的映射队列类吗？                                

● 就绪队列的本质

  就绪队列是 OS 内核维护的数据结构，用来存放所有"可以跑但暂时没拿到
  CPU"的线程。它不在 Java 层面，Java 代码无法访问，也没有对应的映射类。

  它在哪一层

  Java 层        Thread.start()  →  只是告诉 JVM "创建线程"
                      ↓
  JVM 层         调用 OS 的系统调用（pthread_create / CreateThread）
                      ↓
  OS 内核层      创建内核线程 → 放入就绪队列 → 由调度器分配 CPU

  Java 线程是 1:1 映射到 OS 内核线程的（HotSpot JVM 的实现）。线程的调度完全交给
   OS，JVM 不参与调度决策。

  就绪队列的实际结构

  它也不是一个简单的"队列"，不同 OS 实现不同：

  ┌─────────────┬────────────────┬──────────────────────────────────────────┐
  │     OS      │     调度器     │          就绪队列的实际数据结构          │
  ├─────────────┼────────────────┼──────────────────────────────────────────┤
  │ Linux (CFS) │ 完全公平调度器 │ 红黑树，按虚拟运行时间排序，最左节点优先 │
  ├─────────────┼────────────────┼──────────────────────────────────────────┤
  │ Windows     │ 多级反馈队列   │ 32 个优先级队列数组，每个优先级一个链表  │
  └─────────────┴────────────────┴──────────────────────────────────────────┘

  叫"队列"只是一个概念性的统称，实际上是更复杂的数据结构来保证调度效率。

  Java 中能控制什么

  Java 无法直接操作就绪队列，只能通过有限的 API 间接影响 OS 的调度行为：

  thread.setPriority(10);   // 建议 OS 提高优先级，OS 可以忽略
  Thread.yield();           // 建议 OS "我愿意让出 CPU"，OS 可以忽略
  Thread.sleep(1);          // 强制离开就绪队列，进入定时等待

  注意前两个只是"建议"，OS 有最终决定权。

  一句话总结

  就绪队列是 OS 内核的内部结构，Java 中没有对应的类，Java 线程从 start()
  那一刻起就交给 OS 管理调度了。

> - synchronized 竞争锁失败 → BLOCKED                                           
  - wait() / park() / sleep() → WAITING / TIMED_WAITING                         
上面的操作后线程又是进入什么队列？                                              

● 核心逻辑

  RUNNABLE 的线程在就绪队列里等 CPU，而 BLOCKED / WAITING
  的线程在各自的等待队列里等条件满足。条件满足后，才回到就绪队列重新排队等 CPU。

  三种情况分别进入什么队列

  1. synchronized 竞争失败 → BLOCKED

  进入该锁对象的 Monitor 的 EntryList（入口队列）

  synchronized(lock) { ... }

  Monitor 内部结构（HotSpot 实现）：
    ├── Owner        —— 当前持有锁的线程（只有 1 个）
    ├── cxq          —— 新来的竞争线程先到这里（栈结构）
    └── EntryList    —— 从 cxq 转移过来，等待竞争锁（链表）

  流程：
    竞争失败 → 进入 cxq → 转移到 EntryList → Owner 释放锁 → 从 EntryList
  中选一个 → RUNNABLE

  2. wait() → WAITING

  进入同一个 Monitor 的 WaitSet（等待集合）

  synchronized(lock) {
      lock.wait();    // 释放锁，进入 WaitSet
  }

  流程：
    wait() → 释放锁 → 进入 WaitSet
                           ↓
                      别的线程调用 lock.notify()
                           ↓
                      从 WaitSet → 移到 EntryList → 重新竞争锁
                           ↓
                      拿到锁 → RUNNABLE

  注意：notify() 后不是直接 RUNNABLE，而是先回到 EntryList 重新竞争锁。

  3. park() → WAITING（ReentrantLock / AQS 体系）

  进入 AQS 的 CLH 队列（这是 Java 层面自己维护的队列）

  ReentrantLock.lock()
      → tryAcquire() 失败
      → 封装成 Node 加入 CLH 队列
      → LockSupport.park() 挂起

  CLH 队列：  head → Node(线程A) → Node(线程B) → Node(线程C) → tail

  流程：
    park() → 挂起，在 CLH 队列中排队
                     ↓
                前驱节点释放锁，调用 unpark()
                     ↓
                被唤醒 → RUNNABLE

  4. sleep(ms) → TIMED_WAITING

  不进入任何 Java 层面的队列，直接由 OS 的定时器机制管理

  流程：
    sleep(1000) → OS 注册一个定时器 → 线程挂起
                                          ↓
                                     时间到，OS 定时器触发
                                          ↓
                                     放回就绪队列 → RUNNABLE
