CountDownLatch的应用场景:

 最典型的就是**"等初始化完成，然后开始干活，但初始化线程不能停"**。

  1. 配置中心 / 注册中心连接

  启动线程 → 连接 Nacos → 拉取配置完成 → countDown
                                           ↓
  主线程 await() → 拿到配置 → 启动 Web 服务

  连接线程继续运行：监听配置变更、心跳续约...（不能停）

  如果用 join，Web 服务永远启动不了，因为连接线程一直在跑。

  2. 线程池 / 后台服务预热

  线程池启动 → 核心线程创建完毕 → countDown
                                   ↓
  主线程 await() → 开始接收请求

  线程池继续运行：等待任务提交...（不能停）

  比如 Tomcat 启动时，要等 Connector 初始化好才能对外服务，但 Connector 线程本身要一直监听端口。

  3. 缓存预热

  缓存线程 → 从 DB 加载热点数据到 Redis → countDown
                                            ↓
  主线程 await() → 开放接口给用户访问

  缓存线程继续运行：定时刷新缓存...（不能停）

  如果不等预热完就开放接口，用户请求全部穿透到 DB。

  4. MQ 消费者就绪

  消费者线程 → 连接 Kafka → 订阅 topic 成功 → countDown
                                                ↓
  主线程 await() → 发布服务到注册中心（告诉别人"我准备好了"）

  消费者线程继续运行：持续消费消息...（不能停）

  这些场景的共同模式

  长期运行的线程：  ──[初始化]──countDown()──[持续运行，永不结束]──→
                                ↓
  等待方：          ──await()───→ 确认就绪，继续后续逻辑

  核心特征：你关心的是"准备好了没"，而不是"干完了没"。 join 只能表达后者。
