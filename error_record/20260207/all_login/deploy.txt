现在你手上有 3 台云主机:
  ┌─────────────────────────────────────────────────────┐
  │  DB 节点   106.55.7.17   →  SQL Server (已在用)        │
  │  主机 A    ?.?.?.?        →  可跑 nginx (你刚说的)      │
  │  主机 B    ?.?.?.?        →  还空着                    │
  └─────────────────────────────────────────────────────┘

  那么一个很自然的企业级分层就出来了,我理解你想往这个方向走:

              用户浏览器
                  │ HTTPS
                  ▼
     ┌──────────────────────────┐
     │  主机 A: nginx           │
     │   ├─ 托管前端 dist 静态资源 │   ← mylogin-front 的产物
     │   └─ /api 反向代理 ──────┐ │   ← 顺便解决跨域,前端不再直连 8888
     └──────────────────────────┘ │
                                   ▼
     ┌──────────────────────────┐
     │  主机 B: Spring Boot 后端  │   ← mylogin-back 的 jar/镜像
     └──────────────────────────┘
                  │ JDBC
                  ▼
          DB 节点 (SQL Server)

  这样 nginx 成为唯一入口,HTTPS 在 nginx 这层终结(证书放这儿),前端是纯静态、后端在内网。这是最主流的企业布局,你 Java 后端应该一看就懂——nginx
  相当于网关 + 静态资源服务器二合一。
