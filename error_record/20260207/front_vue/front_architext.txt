 前端宏观架构（鸟瞰视角）

  一、它在系统里扮演什么角色

  一个 SPA（单页应用）：浏览器只加载一次 HTML，之后所有"页面切换"都在浏览器里完成，靠 JS 改 DOM；和后端只通过 JSON HTTP 接口通信，不再由后端渲染页面。

  ▎ Java 类比：以前 Spring MVC 返回 JSP/Thymeleaf 是"服务端拼好页面发过来"；现在后端退化成纯 REST API，前端是一个独立的客户端程序，恰好运行在浏览器里。

  三、整体形状（分层）

  ┌──────────────────────────────────────────────────┐
  │  入口层      main.js + App.vue                    │  装配 + 挂载
  ├──────────────────────────────────────────────────┤
  │  路由层      router/                              │  URL → 组件 + 访问控制
  ├──────────────────────────────────────────────────┤
  │  视图层      views/   (LoginView / Dashboard...)  │  页面 = 组件树
  ├──────────────────────────────────────────────────┤
  │  状态层      stores/  (Pinia)                     │  跨组件共享的"全局变量"
  ├──────────────────────────────────────────────────┤
  │  通信层      api/     (Axios 实例 + 拦截器)        │  和后端对话的唯一通道
  └──────────────────────────────────────────────────┘
                            │
                            ▼
                    后端 REST API (8888)

  层和层之间的方向是单向自顶向下的：视图调状态、状态调通信、通信打后端。反过来不依赖。

  ▎ Java 类比：很像 Controller → Service → Repository 的分层，只不过这里少了 Service 层（store 同时承担了状态 + 业务编排）。

  ---
  四、几个架构级的设计选择

  1. 客户端路由（Client-side Routing）
  URL 切换不发请求给后端，而是由 Vue Router 在浏览器里匹配 → 换组件。后端的 URL 命名空间从此只剩 /api/**。
  2. 集中式状态 + 组件本地状态混用
  全局共享的（如登录用户）放 Pinia store；只属于某个页面的（如表单输入）放组件自己的 ref。
  3. HTTP 通信收敛到单一入口
  全应用只有一个 Axios 实例，所有请求都从这里出。所以跨域、超时、Cookie 携带、统一错误处理（如 401）只需配一次。
  4. 认证态采用"服务端为真，前端镜像"模型
  登录态的"真相"在后端 Session，前端只是缓存一份用户名用于渲染。这决定了刷新页面这一动作必须由路由守卫主动向后端确认 Session 是否还活着。
  5. 构建工具是 Vite
  开发期靠浏览器原生 ES Module 即时加载，无打包；生产期才打包。

  ▎ Java 类比：开发期像 Spring DevTools 热重载，启动飞快；生产期像 mvn package 产出可部署的静态产物（一堆 HTML/JS/CSS）。

  ---
  五、运行时的"三个世界"

  理解前端架构最容易卡的一点：浏览器里同时运行着三套互不相同的状态，它们之间需要协调：

  ┌─────────────────────────┐
  │ ① URL / 浏览器历史栈     │  ← 用户点链接、按后退键、刷新
  └─────────────────────────┘
              ↕ Vue Router 维护一致性
  ┌─────────────────────────┐
  │ ② Pinia 内存状态         │  ← JS 变量，刷新页面就丢
  └─────────────────────────┘
              ↕ 路由守卫 + Axios 协调
  ┌─────────────────────────┐
  │ ③ 后端 Session (Cookie) │  ← 跨刷新存活的"真相"
  └─────────────────────────┘

  整个前端架构的关键工作，本质上就是让这三者保持一致：

  - URL 变了 → 路由层决定让哪个组件上场，并检查权限
  - 内存丢了（刷新）→ 找后端确认 Session，重建内存
  - 后端说没登录（401）→ 清内存 + 改 URL 跳登录页

  ---
  六、一句话总结

  ▎ 一个 Vite 构建的 Vue 3 SPA，以 Pinia 集中托管全局状态、Vue Router 做客户端路由、Axios 单实例承担所有后端通信；架构核心议题是协调"URL / 内存状态 / 后端
  ▎ Session"三者一致。
