● 权限

  ---
  先把概念分清楚

  权限这个词在前端语境里其实混着两件事：

  认证（Authentication）→ 你是谁？登录了吗？
  授权（Authorization）  → 你能做什么？有没有资格？

  守卫上一节讲的主要是认证——"你有没有登录"。权限讲的是授权——"你登录了，但你能不能看这个页面、做这个操作"。

  ---
  服务端时代的权限

  和守卫一样，服务端时代权限检查是物理关卡：

  GET /admin/users
  → 服务器：你是谁？admin 还是普通用户？
      → 普通用户 → 403 Forbidden，页面根本不给你
      → admin    → 返回数据

  用户拿不到页面，就没有"前端权限"这个问题。

  ---
  前端路由之后，权限变成了什么

  SPA 把所有代码打包进了一个 JS 文件，所有页面的组件代码都在用户浏览器里。

  普通用户打开页面
  → 下载 main.js（里面包含 AdminView 的代码）
  → 守卫拦住了，不让他跳转到 /admin
  → 但 AdminView 的代码他已经拿到了

  守卫只是不渲染那个组件，代码本身藏不住。这是前端权限的先天局限——前端能控制的只是"显示什么"，藏不住代码，更藏不住数据。

  ---
  前端权限实际在控制什么

  分三个层次：

  第一层：路由层

  /admin  → meta: { role: 'admin' }

  守卫检查：当前用户的角色是不是 admin？
      → 不是 → 跳转到 /403 或 /dashboard
      → 是   → 放行

  控制的是能不能进入这个页面。

  第二层：页面内元素

  <el-button v-if="user.role === 'admin'" @click="deleteUser">
    删除用户
  </el-button>

  控制的是页面里的按钮、菜单、操作能不能看到。

  第三层：接口调用

  前端发请求 → 后端验权 → 没权限返回 403

  这一层才是真正有效的，前两层都只是体验。

  ---
  mylogin-front 目前的状态

  现在这个项目的权限是最简单的形态——只有认证，没有授权。

  用户表：
  id | username | password | enabled
                            ↑
                      只有启用/禁用，没有角色字段

  后端代码里也有一行注释说明了这一点：

  // TODO: add role table
  roles: USER  ← 所有人硬编码成同一个角色

  所以现在的逻辑是：登录了就能看所有页面，没登录就什么都看不了。这是权限系统的最简版本，只有"登录/未登录"这一个维度。

  ---
  真正的权限系统长什么样

  如果这个项目要扩展权限，演化路径大概是：

  现在：
      登录 → 能看所有页面

  加角色之后：
      登录 + role=admin  → 能看管理页面
      登录 + role=user   → 只能看普通页面

  加细粒度权限之后：
      登录 + permission=user:delete → 能点删除按钮
      登录 + permission=user:read   → 只能看，不能删

  每一层扩展，前端的工作是控制"显示什么"，后端的工作是控制"给什么数据、允许什么操作"。两者必须同步，否则前端藏起来了按钮，用户直接调接口还是能删。

  ---
  一句话总结

  前端权限控制的是用户看到什么，后端权限控制的是用户能得到什么。前端权限是体验，后端权限是安全，两者职责不同，缺一不可。mylogin-front 目前只有最简单的认证层，授权体系尚未建立。
