 守卫（Navigation Guard）

  ---
  从服务端说起

  服务端路由时代，"你能不能看这个页面"是服务器决定的。

  用户请求 GET /dashboard
  → 服务器先检查：你有没有登录？
      → 没有 → 直接返回 302 跳转到 /login
      → 有   → 执行业务逻辑，返回页面

  这个检查发生在服务器内部，用户看不到，也绕不过。HTTP 响应没回来之前，页面根本不存在。

  ---
  前端路由的天然漏洞

  路由控制权转移到浏览器之后，问题来了。

  前端路由的本质是：JS 读取当前 URL，决定渲染哪个组件。

  但 JS 是跑在用户浏览器里的，用户可以：

  直接在地址栏输入 http://localhost:5173/dashboard
  → Vue Router 收到请求
  → 找到对应组件：DashboardView
  → 渲染
  → 完成

  整个过程没有任何东西拦截。没有登录也能进去。

  服务端路由时代不存在这个问题，因为页面是服务器生成的，服务器有机会在"生成"之前做检查。前端路由把组件都打包到了 JS 里，浏览器已经全部拿到了，根本没有"请求服务器生成页面"这一步。

  ---
  守卫是什么

  守卫就是在路由跳转真正发生之前插入一个检查点。

  用户触发跳转（点链接、输入URL、调用 router.push）
          ↓
      ← 守卫 →         ← 这里是检查点
          ↓
      路由跳转发生
          ↓
      组件渲染

  守卫可以在这个检查点做三件事：

  放行   → 跳转正常发生
  重定向 → 去别的地方
  取消   → 哪也不去

  ---
  守卫解决的本质问题

  服务端路由，权限检查是物理上的必经之路——HTTP 请求必须经过服务器，服务器在那里拦。

  前端路由，路由跳转只是 JS 函数调用，没有物理关卡。守卫就是人为制造的关卡，强行插在跳转流程里。

  服务端：请求 → [服务器检查] → 响应       物理关卡，绕不过
  前端：  跳转 → [守卫检查]   → 渲染       逻辑关卡，靠约定

  注意"靠约定"这三个字。守卫是 Vue Router 框架保证会执行的，但它的本质是 JS 代码，运行在用户的浏览器里。这引出了一个重要认知——

  ---
  守卫不是真正的安全屏障

  守卫只能保护用户体验，不能保护数据。

  攻击者绕过守卫的方式：打开浏览器控制台
  → 直接调用 API：fetch('http://localhost:8888/api/users')
  → 守卫完全不知道，拦不住

  守卫拦的是"页面跳转"，不是"数据请求"。

  真正的权限保护在后端：

  前端守卫  → 保护用户不看到他不该看的页面   （体验层）
  后端接口  → 保护数据不被未授权的请求拿走   （安全层）

  两者缺一不可，但职责不同。mylogin-front 的守卫做的是体验层的事，真正的安全靠 Spring Security 在后端拦截。

  ---
  一句话总结

  守卫是前端为了弥补"路由跳转没有物理关卡"这个先天缺陷而引入的逻辑检查点，它能保证用户看到正确的页面，但不能替代后端的权限验证。
