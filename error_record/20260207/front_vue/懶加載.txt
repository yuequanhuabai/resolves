● 懒加载

  ---
  从打包说起

  前端路由能工作，依赖一个前提：所有页面的组件代码都在浏览器里。

  npm run build 做的事情，就是把你写的所有 Vue 组件、所有依赖，打包成一个 JS 文件：

  src/
  ├── views/LoginView.vue
  ├── views/DashboardView.vue
  └── views/AsyncDemoView.vue
          ↓ 打包
  dist/
  └── assets/main.abc123.js   ← 所有东西都在这里

  用户打开网站，第一件事是下载这个 main.js。

  项目小的时候没问题。但项目长大之后：

  main.js = 登录页 + 仪表盘 + 用户管理 + 报表 + 设置 + 帮助中心 + ...
          = 几 MB 甚至十几 MB

  用户打开首页，只是想登录，却要把报表、帮助中心的代码全部下载完才能看到页面。在网速慢的环境下，白屏可能长达数秒。

  ---
  懒加载是什么

  懒加载的思路是：用户没去的页面，代码先不下载。

  用户打开网站
  → 只下载首页需要的代码
  → 用户跳转到仪表盘
  → 这时候才下载仪表盘的代码
  → 用户跳转到报表页
  → 这时候才下载报表的代码

  代码被拆成多个小块，按需加载。这叫 Code Splitting（代码分割），懒加载是它的使用方式。

  ---
  路由为什么是天然的分割点

  每个路由对应一个页面，每个页面对应一组组件。用户在同一时刻只能看一个页面。

  所以路由是最自然的分割边界——按路由分割，用户跳转到哪里才加载哪里的代码。

  mylogin-front 里的写法：

  // 不用懒加载：
  import DashboardView from '../views/DashboardView.vue'
  { path: '/dashboard', component: DashboardView }
  // 打包时 DashboardView 进入 main.js，首屏就下载

  // 用懒加载：
  { path: '/dashboard', component: () => import('../views/DashboardView.vue') }
  // 打包时 DashboardView 单独成一个文件，跳转时才下载

  一个箭头函数 () => import(...) 的区别，构建工具（Vite/Webpack）看到这个写法，就知道要把这个组件单独打包成一个文件。

  ---
  构建产物的变化

  不用懒加载：
  dist/assets/main.abc123.js        ← 2MB，所有页面都在里面

  用了懒加载：
  dist/assets/main.abc123.js        ← 200KB，只有框架和首页
  dist/assets/Dashboard.xyz456.js   ← 单独的仪表盘代码
  dist/assets/AsyncDemo.def789.js   ← 单独的异步演示代码

  用户首次访问只下载 main.js，首屏更快。

  ---
  懒加载带来的新问题

  天下没有免费的午餐。懒加载解决了首屏速度，引入了新的体验问题：

  跳转时的等待

  用户点击"进入仪表盘"
  → 路由开始跳转
  → 发现 DashboardView 还没下载
  → 发起网络请求下载 Dashboard.js
  → 下载完成
  → 渲染组件
  → 用户看到页面

  这中间有一段等待时间。网速好时几十毫秒感知不到，网速差时可能要一两秒，期间页面没有任何反馈，用户会以为点击没反应。

  所以引入懒加载之后，通常还需要配套：

  加载状态提示  → 跳转中显示进度条（如 NProgress）
  错误处理      → 下载失败时给用户反馈
  预加载        → 提前猜测用户下一步要去哪，偷偷预下载

  这些都是懒加载带来的连带复杂度。

  ---
  一句话总结

  懒加载是用"跳转时多一次网络请求"换"首屏少下载很多代码"，路由是它最自然的分割边界。代价是跳转体验需要额外处理，否则用户会感到页面"卡了一下"。
