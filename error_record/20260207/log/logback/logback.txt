● 当前项目 logback 配置概览

  你的配置结构很清晰，主要特点：

  - 5 个分级文件 Appender：ALL / INFO / DEBUG / WARN / ERROR，各自独立文件
  - 全部使用异步写入 (AsyncAppender)，提升性能
  - 滚动策略：TimeBasedRollingPolicy + SizeAndTimeBasedFNATP，按天+按大小滚动
  - 归档格式：全部使用 .gz 压缩（如第 37 行 %d{yyyy-MM-dd}.%i.log.gz）
  - 3 个 Spring Profile：local、dev、test/stage/prod

  ---
  关于 logback 支持的压缩格式

  Logback 根据 fileNamePattern
  的文件后缀自动判断是否压缩以及使用哪种压缩算法。原生只支持两种：

  ┌──────┬──────────┬────────────────────┐
  │ 后缀 │ 压缩算法 │        说明        │
  ├──────┼──────────┼────────────────────┤
  │ .gz  │ GZIP     │ 当前项目使用的方式 │
  ├──────┼──────────┼────────────────────┤
  │ .zip │ ZIP      │ 另一种支持的格式   │
  └──────┴──────────┴────────────────────┘

  能否改为 .tar.gz？

  不能。 Logback 不支持 .tar.gz。

  原因：
  - .tar.gz 是两步操作：先 tar 打包（多文件归为一个），再 gzip 压缩
  - Logback 的滚动归档是单文件压缩（一个日志文件 →
  一个压缩文件），不涉及"打包"步骤
  - Logback 源码中只识别 .gz 和 .zip 两种后缀，遇到 .tar.gz
  会被当作不压缩处理（或者只识别最后的 .gz 但产出的实际上是 gzip 文件而非 tar.gz
   文件）

  如果你将 fileNamePattern 改为 .tar.gz，实际效果是：文件名虽然叫 
  .tar.gz，但内容本质上只是 .gz（不是真正的 tar 归档），这会造成误导。

  建议

  - 如果只是需要压缩，继续用 .gz 即可，这是最佳实践
  - 如果确实需要 .tar.gz 归档（比如运维要求），可以通过外部脚本/cron 
  任务对已归档的 .gz 文件做二次打包

● 是的，你的理解完全正确。

  Logback 只看最后一个后缀来决定压缩方式。对于 .tar.gz：

  - Logback 识别到最后的 .gz → 执行 GZIP 压缩
  - .tar 部分被当作文件名的一部分，没有任何 tar 打包操作

  IHUB-PAP-ALL.applog.2026-03-11.0.log.tar.gz

  # 但实际内容等价于
  IHUB-PAP-ALL.applog.2026-03-11.0.log.tar.gz  ← 本质就是一个 gzip 文件

  这会带来一个实际问题：运维或开发人员拿到这个文件后，会尝试用 tar -xzf
  来解压，结果会报错，因为它根本不是 tar 归档。只能用 gzip -d 或 gunzip
  解压，解压后得到的文件名还带着 .tar（即 xxx.log.tar），非常容易让人困惑。

  所以建议不要用 .tar.gz，直接用 .gz
  就好，这也是你当前配置已经在用的方式，没有问题。

> 我觉得归档日志这个功能很通用啊，为什么logback没有呢？                         

● 这个问题很好，原因其实是设计定位不同。

  Logback 不做 tar 打包的核心原因

  Logback 的归档是"一对一"的——一个活跃日志文件滚动后生成一个压缩文件。这个过程中
  根本不存在"多个文件合并"的场景，而 tar 的本质就是把多个文件打包成一个。

  简单说：

  logback 滚动归档的模型：
    IHUB_ALL.log  →  2026-03-11.0.log.gz   (1 对 1)

  tar 的使用场景：
    file1 + file2 + file3  →  archive.tar.gz  (多 对 1)

  一个日志文件压缩成一个 .gz，tar 这层打包完全是多余的。

  更深层的原因

  1. 职责分离——Logback 定位是日志框架，不是归档/备份工具。它只负责"写日志、滚动
  、单文件压缩"，把日志的归档管理（合并、转移、清理）留给运维工具去做（如
  logrotate、cron 脚本、日志收集平台 ELK 等）
  2. 性能考量——日志滚动发生在应用运行时，gzip 单文件压缩已经很快，如果再加 tar
  打包逻辑，会增加复杂度和 I/O 开销，对应用性能没有好处
  3. gz 本身就能独立压缩单文件——gzip 不像 zip 那样天然支持多文件，但压缩单文件时
   gzip 比 zip 更高效。单文件场景下 .gz 就是最合理的选择，不需要 tar 这层

  如果运维确实要求 .tar.gz

  那属于日志归档管理的范畴，通常的做法是：

  # 每天凌晨用 cron 把前一天的多个 .gz 打成一个 tar 包
  0 2 * * * tar -cf /LOG/2026-03-10.tar /LOG/*.2026-03-10.*.gz && rm
  /LOG/*.2026-03-10.*.gz

  这也印证了——打包是运维的事，不是日志框架的事。

  方法一：临时改小 maxFileSize（最简单）

  把 maxFileSize 改成极小值，然后随便触发几条日志就会滚动：

  <maxFileSize>1KB</maxFileSize>

  重启应用，请求几个接口产生日志，就会立即触发归档。验证完改回去即可。

  方法二：写代码手动调用 rollover()

  写一个临时的测试接口：

  @GetMapping("/test-rollover")
  public String testRollover() {
      LoggerContext context = (LoggerContext) LoggerFactory.getILoggerFactory();
      for (Logger logger : context.getLoggerList()) {
          for (Iterator<Appender<ILoggingEvent>> it =
  logger.iteratorForAppenders(); it.hasNext(); ) {
              Appender<ILoggingEvent> appender = it.next();
              if (appender instanceof RollingFileAppender<ILoggingEvent>
  rolling) {
                  rolling.rollover();  // 手动触发归档
              }
          }
      }
      return "rollover triggered";
  }

  调用一次就会对所有 RollingFileAppender 执行一次归档。

  在 logback-spring.xml 中加上：

  <jmxConfigurator />

  然后通过 JConsole 连上应用，可以操作 logback 的
  MBean，但这个主要是重载配置，不能直接触发 rollover。

> 我还有个问题，logback生成的日志目录如果不存在，它会自动创建吗                 

● 会的。 Logback 会自动创建日志文件及其所有父目录。

  无论是 <file> 指定的活跃日志路径，还是 <fileNamePattern>
  指定的归档路径，如果目录不存在，logback 都会在第一次写入时自动递归创建。

  所以你当前配置中的 logs/ 和 logs/history/ 目录都不需要手动创建，应用启动后
  logback 会自动处理。

> 还有一个问题，如果IHUB_ALL                                                    
..info,debug,warn,error.log-date中有空文件，它也会被归档压缩吗                  

● 不会。 你配置了 notifempty，它的作用就是：如果活跃日志文件为空，直接跳过轮转，
  根本不会产生对应的 -date 文件。

  举个实际场景，比如生产环境 root level 是 INFO，DEBUG 日志没有输出：

  IHUB_ALL.log      → 有内容 → 轮转 → IHUB_ALL.log-20260311 ✓
  IHUB_INFO.log     → 有内容 → 轮转 → IHUB_INFO.log-20260311 ✓
  IHUB_DEBUG.log    → 空文件 → notifempty 跳过，不产生 -date 文件 ✗
  IHUB_WARN.log     → 有内容 → 轮转 → IHUB_WARN.log-20260311 ✓
  IHUB_ERROR.log    → 空文件 → notifempty 跳过，不产生 -date 文件 ✗

  然后 postrotate 的 find -name "IHUB_*.log-20260311" 只会找到 3 个文件，最终
  tar.gz 里也只有这 3 个：

  prod-01.IHUB-PAP.applog.20260311.tar.gz
  ├── IHUB_ALL.log-20260311
  ├── IHUB_INFO.log-20260311
  └── IHUB_WARN.log-20260311

  所以不用担心空文件被打进归档。
