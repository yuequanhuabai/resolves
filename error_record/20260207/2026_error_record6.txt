  RT 了）。复杂度从前端搬回了后端。
  2. 轮换 + 多标签页的潜在摩擦——这是 rotation 的固有代价：如果你开两个标签页、AT 几乎同时过期，两边可能拿同一张 RT 去刷新，第二个会撞上"已 used"→
  触发重放检测 → 误判为泄露，把人踢下线。前端的 isRefreshing 队列只能挡住"同一页内"的并发，挡不住跨标签页。生产级实现常加一个短宽限期（旧 RT
  在几秒内重复使用不算泄露）来缓解。
  3. Cookie 的 path=/api/auth/refresh（AuthController:95）——作用域收得很窄，RT Cookie
  只在调刷新接口时才发送，平时的业务请求根本不带它，进一步缩小暴露面。设计很讲究。
  4. secure(false) 是给本地 HTTP 用的（AuthController:93），注释已写明生产要改 true（仅 HTTPS 传输）。
  5. logout 现在是"真登出"——revokeByUsername 删光该用户所有 RT（AuthController:79），即便 RT 没过期也立即失效。这正是有状态带来的能力，无状态版的
  logout 只能空喊。
