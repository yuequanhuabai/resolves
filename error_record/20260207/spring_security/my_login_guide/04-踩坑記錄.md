# 04 — 踩坑記錄與疑問速查

> 本文檔回答：**「執行 03 文檔時遇到報錯/疑惑怎麼辦？」**
>
> 所有條目按 03 文檔的階段順序排列。每一條的格式是：
> - **症狀**：你會看到什麼現象
> - **根因**：為什麼會這樣
> - **解法**：怎麼修
> - **記憶點**：以後怎麼一眼識別

## 環境類問題（階段 1）

### ❌ `java -version` 顯示的是 8 / 11，不是 17

- **症狀**：命令行 `java -version` 輸出的是低版本
- **根因**：系統裝了多版本 JDK，`PATH` 裏優先級錯了
- **解法**：
  1. Windows 控制面板 → 環境變量 → `JAVA_HOME` 改指向 JDK 17 的安裝目錄
  2. 確保 `PATH` 中 `%JAVA_HOME%\bin` 在所有其他 java 路徑之前
  3. **重新打開**命令行（環境變量改了之後，已開的窗口不會自動更新）
- **記憶點**：環境變量改完一定要新開終端

### ❌ Maven 下載依賴特別慢（拉了半小時還沒完）

- **症狀**：第一次導入項目卡在「Resolving dependencies」
- **根因**：默認從 Maven 中央倉庫拉，國內網絡訪問慢
- **解法**：在 `~/.m2/settings.xml` 配阿里雲鏡像
  ```xml
  <mirror>
    <id>aliyun</id>
    <mirrorOf>central</mirrorOf>
    <url>https://maven.aliyun.com/repository/public</url>
  </mirror>
  ```
- **記憶點**：Maven 慢就改 `settings.xml`，不要動 `pom.xml`

### ❌ 數據庫連不上（DBeaver 都連不上）

- **症狀**：DBeaver 點 Test Connection 報錯
- **可能原因清單**（一個個排查）：
  1. SQL Server 服務沒啟動（Windows 服務管理器看 SQL Server (XXX)）
  2. SQL Server 沒開啟 TCP/IP 協議（SQL Server 配置管理器 → 網絡配置 → 啟用 TCP/IP，**改了要重啟服務**）
  3. 防火牆沒放行 1433 端口
  4. 用戶名密碼錯
  5. SQL Server 沒啟用「混合身份驗證」（純 Windows 認證連不上 sa）
- **解法**：先確認 DBeaver 連得通，這是後續一切的前提

## 項目骨架類（階段 2）

### ❌ 項目啟動時報 `Failed to configure a DataSource`

- **症狀**：啟動異常 `Failed to configure a DataSource: 'url' attribute is not specified...`
- **根因**：你選了 Spring Data JPA 依賴，但 `application.yml` 沒配數據源
- **解法**：兩種選擇：
  1. **推薦**：直接進階段 3 把數據源配上
  2. 臨時：在啟動類註解改成 `@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})`，等配好數據源再去掉
- **記憶點**：這個錯誤 = 「你少配了一個東西」，不是 bug

### ❌ application.properties 和 application.yml 同時存在

- **症狀**：怎麼改 yml 都不生效
- **根因**：Initializr 默認生成的是 `.properties`，你又新建了 `.yml`，**`.properties` 優先級更高**
- **解法**：刪掉 `application.properties`，只保留 `application.yml`
- **記憶點**：兩個文件只能存在一個

### ❌ 訪問 localhost:8080 看到 Spring Security 默認登錄頁

- **症狀**：項目啟動成功，訪問首頁出現一個寫著 `Please sign in` 的英文登錄頁，要密碼
- **根因**：你還沒寫自己的 SecurityConfig，Spring Security 自動配置了一個默認的
  - 默認用戶名：`user`
  - 默認密碼：**啟動日誌裏會打印**，類似 `Using generated security password: 9c1234ab-...`
- **解法**：這是正常的，配完階段 4 的 SecurityConfig 後它就沒了
- **記憶點**：看到陌生英文登錄頁 = 你還沒接管 Spring Security

## 數據層類（階段 3）

### ❌ JPA 啟動報 `Unknown table` / `Table doesn't exist`

- **症狀**：啟動報找不到 sys_user 表
- **可能原因**：
  1. 沒在數據庫真正建表（先去 DBeaver 確認）
  2. `application.yml` 裏連的是錯的數據庫名（`databaseName=` 寫錯）
  3. `@Table(name="sys_user")` 寫錯成 `sys_users` 之類
- **解法**：DBeaver 連到應用配的同一個庫，執行 `SELECT * FROM sys_user`，能查出來就說明配置沒錯

### ❌ JPA 自動把表結構改了 / 把表刪了

- **症狀**：啟動後表沒了，或者表結構被改成奇怪的樣子
- **根因**：`spring.jpa.hibernate.ddl-auto` 配成了 `create` / `create-drop` / `update`
- **解法**：**配成 `none`**。讓 Hibernate 不要碰你的表結構
- **記憶點**：學習階段就用 `none`，自己控制 SQL 最清楚

### ❌ Entity 上的註解報紅 / 找不到 `@Entity`

- **症狀**：IDE 提示 `Cannot resolve symbol 'Entity'`
- **根因**：Spring Boot 3 用 **Jakarta EE**，不是 Java EE
- **解法**：import 寫 `jakarta.persistence.Entity`，**不是 `javax.persistence.Entity`**
- **記憶點**：Spring Boot 3 = Jakarta；Spring Boot 2 = javax

### ❌ Lombok 的 @Data 不生效（getter/setter 找不到）

- **症狀**：`sysUser.getUsername()` 報紅，IDE 找不到方法
- **根因**：IDE 沒裝 Lombok 插件，或者沒開「啟用註解處理器」
- **解法**：
  1. IDEA：Settings → Plugins → 搜 Lombok → 安裝重啟
  2. IDEA：Settings → Build → Compiler → Annotation Processors → 勾上 Enable annotation processing
- **記憶點**：Lombok 是「編譯期生成代碼」，IDE 必須能看懂

### ❌ SQL Server 連接報「encrypt」相關錯誤

- **症狀**：`The driver could not establish a secure connection to SQL Server by using SSL encryption`
- **根因**：mssql-jdbc 12+ 默認要求加密，但你的 SQL Server 沒裝有效證書
- **解法**：URL 裏加 `encrypt=true;trustServerCertificate=true`（demo 階段可接受，生產環境要配真證書）
- **記憶點**：本地 demo 看到 SSL/encrypt 錯誤就加這倆參數

## Spring Security 類問題（階段 4，最容易卡住）

### ❌ 登錄永遠失敗（密碼明明是對的）

- **症狀**：用 admin / 123456 登錄被打回 `/login?error`
- **最常見根因**：**DataInitializer 寫入的密碼沒加密**
  - 直接 `setPassword("123456")` ← 錯！
  - 必須 `setPassword(passwordEncoder.encode("123456"))` ← 對
- **驗證方法**：去數據庫看 `sys_user.password`，正確的應該是 `$2a$10$...` 開頭的 60 字符長串，不是明文
- **修法**：刪掉表裏的舊數據，改正代碼，重啟讓 DataInitializer 重新插一次
- **記憶點**：BCrypt 哈希 = `$2a$` 開頭的 60 字符串

### ❌ 登錄成功但無限跳轉

- **症狀**：登錄成功後跳 `/dashboard`，又被踢回 `/login`，反覆循環
- **可能根因**：
  1. `/dashboard` 沒有對應的 Controller 方法（404 觸發了某些奇怪行為）
  2. SecurityConfig 把 `/dashboard` 也設了 `permitAll()`，但是又用 `defaultSuccessUrl` 指向它
  3. Session 配置出問題（極少見）
- **排查**：先把 dashboard 改成最簡單的 `return "login";`，看是否還循環。如果不循環，問題在 dashboard 視圖
- **記憶點**：循環跳轉 = 認證和授權配置打架了

### ❌ 訪問 dashboard 拋 `Cannot resolve view 'dashboard'`

- **症狀**：登錄成功後白頁 / 500 錯誤
- **根因**：
  1. `dashboard.html` 沒放在 `src/main/resources/templates/` 下
  2. 文件名拼錯（如寫成 `Dashboard.html`，注意大小寫）
  3. Thymeleaf 依賴沒加（pom.xml 缺 `spring-boot-starter-thymeleaf`）
- **解法**：對應檢查，注意 Linux/Mac 文件名大小寫敏感
- **記憶點**：Controller 返回字符串 `"dashboard"` ↔ `templates/dashboard.html`

### ❌ POST `/login` 報 403 Forbidden

- **症狀**：表單提交後直接 403
- **根因**：Spring Security 默認開啟 CSRF 保護，表單沒帶 CSRF token
- **解法**：兩種：
  1. **推薦**：用 Thymeleaf 的 `th:action="@{/login}"`（會自動注入 CSRF token）
  2. 不推薦：在 SecurityConfig 加 `.csrf(csrf -> csrf.disable())`（生產絕不能這麼幹）
- **記憶點**：`th:action` 不只是 URL，還會自動帶 CSRF token

### ❌ 退出登錄報 405 Method Not Allowed

- **症狀**：點退出按鈕報「請求方法不支持 GET」
- **根因**：用了 `<a href="/logout">`，但 Spring Security 默認只接受 POST
- **解法**：把退出做成一個 form：
  ```html
  <form th:action="@{/logout}" method="post"><button>退出</button></form>
  ```
- **記憶點**：退出登錄必須 POST

### ❌ 啟動報 `There is no PasswordEncoder mapped for the id "null"`

- **症狀**：登錄時拋這個異常
- **根因**：你寫了 SecurityConfig 但沒註冊 `PasswordEncoder` Bean
- **解法**：在 SecurityConfig 加 `@Bean public PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); }`
- **記憶點**：用 BCrypt 必須註冊 PasswordEncoder Bean

### ❌ `WebSecurityConfigurerAdapter` 已棄用

- **症狀**：你網上抄的代碼讓你 `extends WebSecurityConfigurerAdapter`，IDE 提示廢棄
- **根因**：Spring Security 5.7+ 已廢棄這種寫法，6.0 完全移除
- **解法**：用新寫法 — 註冊一個 `SecurityFilterChain` Bean（見 03 文檔 4.2）
- **記憶點**：Spring Boot 3 / Security 6 寫法 = `SecurityFilterChain` Bean

## 頁面與接口類問題（階段 5）

### ❌ 訪問 `/api/users` 返回 HTML 而不是 JSON

- **症狀**：fetch 拿到的不是 JSON，是登錄頁 HTML
- **根因**：未登錄訪問被 Spring Security 攔截，302 跳轉 `/login`，瀏覽器（fetch 默認跟隨重定向）拿到的是登錄頁
- **解法**：先登錄再訪問。這也是為什麼 users.html 中要判斷 `if (!res.ok) → 跳登錄頁`
- **記憶點**：JSON 接口被攔截後給的是 HTML，不是 401，這是 form login 的特性

### ❌ 接口返回的 JSON 把 password 也吐出來了

- **症狀**：`/api/users` 響應裏看到密碼哈希
- **根因**：直接返回了 `List<SysUser>`，Jackson 把所有字段都序列化了
- **解法**：用 `UserVO`（DTO）顯式指定要返哪些字段
- **記憶點**：**永遠不要直接把 Entity 返給前端**，這條規則要刻在 DNA 裏

### ❌ Thymeleaf 變量在頁面上顯示成原文（如 `${username}`）

- **症狀**：頁面渲染後直接顯示 `${username}` 字符串
- **根因**：HTML 標籤沒加 Thymeleaf 命名空間，或者用了普通 HTML 屬性而不是 `th:*`
- **解法**：
  1. `<html xmlns:th="http://www.thymeleaf.org">`
  2. 用 `th:text="${username}"` 而不是 `${username}` 寫在文本裏
- **記憶點**：Thymeleaf 屬性都帶 `th:` 前綴

### ❌ 修改 HTML 模板後不刷新（必須重啟才看到改動）

- **症狀**：改了 login.html 但瀏覽器看到的還是舊的
- **根因**：Thymeleaf 默認啟用了模板緩存
- **解法**：`application.yml` 加：
  ```yaml
  spring:
    thymeleaf:
      cache: false
  ```
- **記憶點**：開發階段關緩存，生產階段開緩存

### ❌ users.html 的 fetch 一直失敗

- **症狀**：頁面顯示「請求失敗」
- **排查順序**：
  1. F12 Network 看 `/api/users` 請求的狀態碼
  2. 200 但 res.ok 為 false？看返回的 Content-Type 是不是 application/json
  3. 302？說明 Session 沒了（重啟過後端就會這樣，因為內存 Session 丟了）
  4. 500？看後端日誌
- **記憶點**：前端問題優先打開 F12 看網絡請求

## Cookie / Session 類問題（階段 6）

### ❌ JSESSIONID 看不到

- **症狀**：dashboard 上的 Cookie 表格裏沒有 JSESSIONID
- **可能原因**：
  1. 還沒登錄
  2. 用了無痕模式且沒勾「允許所有 Cookie」
  3. 應用部署在 HTTPS 環境但 Cookie 標了 Secure（本地 HTTP 訪問拿不到）
- **解法**：本地 demo 默認情況下不會出這問題，先確認登錄狀態

### ❌ 勾選「記住我」後關閉瀏覽器再開，依然要重新登錄

- **症狀**：Remember-me 不生效
- **可能根因**：
  1. login.html 中 checkbox 的 `name` 不是 `remember-me`（沒和 SecurityConfig 對齊）
  2. SecurityConfig 沒配 `.rememberMe(...)`
  3. 沒勾選 checkbox 就提交了（一個低級但常犯的錯誤）
- **解法**：登錄後去 F12 Application → Cookies 看是否多了 `remember-me` Cookie。沒有 = 後端沒簽發
- **記憶點**：表單字段 name 必須和 `rememberMeParameter` 配置一致

### ❌ Remember-me Cookie 重啟後端後失效

- **症狀**：勾「記住我」登錄；重啟後端；再訪問 `/dashboard` → 還是被踢到登錄頁
- **根因**：**這是默認的 token-based remember-me 機制的特點**
  - Spring Security 默認用「簽名 hash + key」生成 token，不存數據庫
  - 重啟後端時，內存中的 Session 全沒了
  - JSESSIONID 失效後 → 框架嘗試用 remember-me 重建 Session → 但需要 `UserDetailsService` 還能查到該用戶（如果 DataInitializer 重新跑了，該用戶還存在 → 應該能恢復）
  - **如果你看到失效，可能是 Spring Security 版本對配置的解析有差異**
- **解法**：用 `PersistentTokenRepository` 把 token 存到數據庫（進階話題，本 demo 不展開）
- **記憶點**：Token-based 依賴 key 簽名；Persistent-based 依賴數據庫

### ❌ JSESSIONID 看起來「一直在變」

- **症狀**：每次刷新 Session ID 都不一樣
- **可能根因**：
  1. 瀏覽器禁用 Cookie
  2. 登錄成功時 Spring Security 會做 Session Fixation 防護，**新建一個 Session**（這是默認行為，正確）
  3. 後端反覆重啟（每次重啟內存 Session 全清空）
- **記憶點**：登錄前後 Session ID 變一次是正常的；之後同一會話不應該再變

## 通用調試技巧（任何階段都用得上）

### 🔧 看後端日誌的優先級

```
ERROR  ← 必須看
WARN   ← 看
INFO   ← 啟動信息
DEBUG  ← 平時不開，排查疑難雜症時打開
```

打開 Spring Security 的 DEBUG 日誌，能看到「過濾器鏈每一步在做什麼」：

```yaml
logging:
  level:
    org.springframework.security: DEBUG
```

### 🔧 看前端發請求的步驟

1. F12 → Network 面板
2. 勾選「Preserve log」（重定向時不會丟記錄）
3. 點具體請求看：
   - Headers：請求頭、Cookie、響應狀態碼
   - Response：響應體
   - Cookies：當前域名下的所有 Cookie

### 🔧 SQL 是怎麼跑的

`spring.jpa.show-sql: true` 後，控制檯會打印每條 SQL。
進階：加 `format-sql: true` 讓 SQL 排版清晰；加 `org.hibernate.type.descriptor.sql: TRACE` 能看到參數值。

### 🔧 重置法（核武器，謹慎）

當一切都亂了，分階段回退：

1. 數據錯了 → DBeaver 裏 `TRUNCATE TABLE sys_user` → 重啟 → DataInitializer 重灌
2. Session 錯亂 → 瀏覽器 F12 → Application → Clear site data
3. 構建錯了 → IDE 裏 Maven → Lifecycle → clean → 再 install

## 學習過程中的「為什麼」清單

這些問題不是報錯，是「我看不懂為什麼」。先想 30 秒，想不通再看答案。

### Q1：為什麼 `JpaRepository` 不寫實現也能用？

A：Spring Data JPA 在啟動時掃描繼承了 `Repository` 接口的東西，動態生成代理類來實現它。
方法名 `findByXxx` 會被解析為 `WHERE xxx = ?` 的 SQL。這叫「方法名查詢派生」。

### Q2：為什麼 `UserDetailsService` 我寫了但沒地方註冊它？

A：Spring 的依賴注入會自動找到帶 `@Service` 的這個 Bean。Spring Security 的自動配置會自動拿走它，作為認證流程的一部分。**無侵入裝配**是 Spring 的核心魅力。

### Q3：為什麼 `SecurityFilterChain` 是個 Bean 而不是繼承一個類？

A：這是 Spring Security 5.7+ 推崇的「組合優於繼承」風格。每個 Bean 是獨立的配置單元，更容易測試和替換。

### Q4：BCrypt 為什麼能確保安全？

A：
1. **加鹽**：每次加密自動生成隨機鹽，相同明文加密多次結果都不同
2. **慢哈希**：故意設計成計算慢，暴力破解成本極高
3. **不可逆**：哈希函數的單向性

`$2a$10$...` 中的 `10` 是「強度因子」，每+1 計算量翻倍。

### Q5：為什麼登錄要重定向到 `/dashboard`，而不是直接渲染？

A：這是 **POST-Redirect-GET (PRG) 模式**，避免用戶刷新頁面時重複提交表單。Spring Security 的 `defaultSuccessUrl` 就是基於這個模式。

### Q6：Thymeleaf 是「編譯期」還是「運行時」？

A：**運行時**。每次請求到達時，Thymeleaf 引擎讀模板文件 → 解析 → 結合 Model 中的數據生成 HTML → 返回。
這也是為什麼開發階段要關閉 `spring.thymeleaf.cache` —— 不關的話模板被緩存，改了不生效。
