# 01 — 宏觀架構

> 本文檔回答：**這個項目「是什麼」、「由什麼組成」、「為什麼這樣選」。**
> 它不關心具體怎麼寫代碼，只關心整體形態。

---

## 一、項目定位

一個用來**親手體驗「Web 登錄是怎麼工作的」**的最小可運行 Demo。

要回答的核心問題：
1. 用戶在瀏覽器點「登錄」，後端是怎麼校驗的？
2. 登錄成功後，瀏覽器和服務器之間靠什麼維持「我已登錄」這個狀態？
3. 登錄之後再訪問接口（如 `/api/users`），請求是怎麼被識別的？
4. 「7 天免登錄」是怎麼做到的？

不是要做企業級系統，所以**故意不引入** JWT、Redis、OAuth、微服務、前後端分離框架（Vue/React）。

---

## 二、技術選型

| 層 | 技術 | 為什麼選它 |
|---|---|---|
| 構建 | Maven | Spring Boot 生態最主流；用 Gradle 也可以，但 Maven 配置更直觀 |
| 框架 | **Spring Boot 3.2.5** | 自動配置，省去大量 XML / 手動裝配；3.x 對應 Java 17+ |
| Web | spring-boot-starter-web | 內嵌 Tomcat，零配置即可跑 HTTP 服務 |
| 安全 | **spring-boot-starter-security** | 學習目標就是「登錄」本身，Spring Security 是 Java 生態事實標準 |
| 視圖 | **Thymeleaf** | 服務端渲染模板，避免引入 Vue/React 增加心智負擔；可以直接寫 HTML |
| ORM | spring-boot-starter-data-jpa | 不用手寫 SQL，`Repository` 接口自動生成 CRUD |
| 數據庫 | SQL Server (mssql-jdbc) | 沿用作者已有環境；換 MySQL/PostgreSQL 只是改驅動和 URL |
| 工具 | Lombok | `@Data` 一個註解省掉 getter/setter；可選，不裝也能寫 |
| Java | **17+** | Spring Boot 3.x 最低要求 |

 
## 三、模塊劃分（包結構心智模型）

```
com.example.testlogin
├── TestLoginApplication        ← 啟動類（main 方法）
│
├── config/                      ← 配置層：把框架「調成你想要的樣子」
│   └── SecurityConfig           ← Spring Security 規則：誰能訪問什麼、登錄頁在哪
│
├── entity/                      ← 數據庫實體層：和數據庫表一一對應
│   └── SysUser                  ← 對應 sys_user 表
│
├── repository/                  ← 數據訪問層：操作數據庫
│   └── UserRepository           ← 繼承 JpaRepository，自動獲得 CRUD
│
├── service/                     ← 業務/橋接層
│   └── UserDetailsServiceImpl   ← 把「我們的 SysUser」翻譯成「Spring Security 認得的 UserDetails」
│
├── controller/                  ← 接口層：接收 HTTP 請求
│   ├── PageController           ← 返回頁面（HTML），如 /login /dashboard /users
│   ├── UserController           ← 返回數據（JSON），如 GET /api/users
│   └── UserVO                   ← 接口返回值的形狀（DTO）
│
└── init/                        ← 啟動鉤子
    └── DataInitializer          ← 應用啟動後自動寫入 admin/user1/user2 三個測試帳號

resources/
├── application.yml              ← 端口、數據庫連接、JPA 配置
├── templates/                   ← Thymeleaf HTML 模板
│   ├── login.html               ← 登錄頁
│   ├── dashboard.html           ← 登錄後的首頁，展示 Session/Cookie 信息
│   └── users.html               ← 用戶列表頁（演示 fetch 接口時自動帶 Cookie）
└── sql/init.sql                 ← 建庫建表腳本
```

### 為什麼這樣分包？

這套分層遵循的是**經典三層架構 + Spring 生態約定**：

```
HTTP 請求
   │
   ▼
Controller（要不要返回 HTML？要不要返回 JSON？）
   │
   ▼
Service（業務邏輯；本項目薄，僅實現 UserDetailsService）
   │
   ▼
Repository（數據庫操作）
   │
   ▼
Entity（數據庫表的 Java 表示）
```

**Config / Init** 是橫切於這條主鏈的：
- **Config** 配置框架行為（Security 規則、密碼加密器…）
- **Init** 應用啟動完成後執行一次性任務（造測試數據）

## 四、前後端如何交互（兩種模式同時存在）

這個 Demo 故意把**兩種主流的前後端交互方式**都演示了一遍：

### 模式 A：服務端渲染（SSR）— 用於頁面

```
瀏覽器 ── GET /login ──▶ PageController.loginPage()
                              │
                              ▼
                         返回字符串 "login"
                              │
                              ▼
                    Thymeleaf 找到 templates/login.html
                              │
                              ▼
                    把模板渲染成完整 HTML
                              │
瀏覽器 ◀──────  HTML 響應  ─────┘
```

特點：**HTML 在服務器拼好**，瀏覽器拿到就是渲染完成的頁面。

### 模式 B：前後端分離式（局部）— 用於數據

```
瀏覽器加載 users.html（Thymeleaf 模板，但只有殼子）
       │
       ▼
頁面內 <script> 執行 fetch('/api/users')
       │
       ▼  （瀏覽器自動帶上 JSESSIONID Cookie，因為同源）
       ▼
UserController.listUsers()
       │
       ▼
Repository 查庫 → List<UserVO>
       │
       ▼  Spring Boot 自動序列化為 JSON
       ▼
瀏覽器拿到 JSON → JS 動態渲染表格
```

特點：頁面殼子來自 Thymeleaf，但**數據通過 AJAX 異步加載**，更接近現代前端的做法。

> 為什麼要兩種都做？  
> 因為這恰好對應現實開發中的兩個典型場景：登錄頁這種低頻、靜態的用 SSR；用戶列表這種需要動態刷新的用接口。**理解兩者背後的請求/Cookie 流程是一致的**，是這個 Demo 的關鍵教學目標之一。

## 五、認證流程（最關鍵的一張圖）

```
┌─────────────────────────────────────────────────────────────┐
│  ① 訪問受保護資源（如 /dashboard）— 未登錄                       │
│                                                              │
│  Browser ───── GET /dashboard ─────▶ Spring Security Filter  │
│         ◀────  302 重定向到 /login ─────                      │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  ② 提交登錄表單                                                │
│                                                              │
│  Browser ── POST /login (username, password, remember-me)    │
│                          ─────▶ UsernamePasswordAuthFilter   │
│                                       │                      │
│                                       ▼                      │
│                          UserDetailsServiceImpl              │
│                          .loadUserByUsername(username)       │
│                                       │                      │
│                                       ▼                      │
│                                  查 sys_user 表               │
│                                       │                      │
│                                       ▼                      │
│                          BCryptPasswordEncoder.matches()     │
│                          （比對輸入明文 與 數據庫密文）          │
│                                       │                      │
│              ┌────────── 成功 ────────┴────── 失敗 ──────┐    │
│              ▼                                         ▼    │
│  創建 HttpSession，                           302 → /login  │
│  把 Authentication 存入                          ?error      │
│  SecurityContext，                                          │
│  種下 JSESSIONID Cookie                                      │
│              │                                              │
│              ▼                                              │
│  302 → /dashboard                                           │
│  （勾選 remember-me 還會多種一個 7 天 Cookie）                  │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  ③ 後續請求（包括 /dashboard、/api/users）                     │
│                                                              │
│  Browser ─── 自動帶上 JSESSIONID Cookie ───▶ Filter          │
│                          │                                   │
│                          ▼                                   │
│                  從 Session 中還原 Authentication             │
│                          │                                   │
│                          ▼                                   │
│                     正常響應業務數據                           │
└─────────────────────────────────────────────────────────────┘
```

### 三個必須記住的關鍵概念

1. **JSESSIONID** — 由 Servlet 容器（Tomcat）發的 Cookie，是「服務端 Session」的鑰匙。瀏覽器關閉就消失。
2. **SecurityContext** — Spring Security 用來保存「當前是誰」的容器。它默認被存到 HttpSession 裏。
3. **remember-me Cookie** — 額外簽發的長效 Cookie，瀏覽器關了還在。下次訪問時，即使 JSESSIONID 沒了，Spring Security 也能用它重建 Session。

## 六、數據流（從一個 HTTP 請求看穿全棧）

以 **「登錄成功後，訪問 `/api/users`」** 為例：

```
1. 用戶點「用戶列表」鏈接
   ▼
2. 瀏覽器加載 users.html（PageController + Thymeleaf 渲染殼子）
   ▼
3. 頁面 JS 執行 fetch('/api/users')
   ▼  瀏覽器自動帶 Cookie: JSESSIONID=xxx
   ▼
4. 請求到達 Tomcat → Spring Security Filter Chain
   ▼
5. 從 JSESSIONID 找到 HttpSession → 還原 SecurityContext → 認證通過
   ▼
6. DispatcherServlet 路由到 UserController.listUsers()
   ▼
7. UserController 調 userRepository.findAll()
   ▼
8. Spring Data JPA 生成 SQL: SELECT * FROM sys_user
   ▼
9. JDBC 通過 mssql-jdbc 連到 SQL Server，拿到結果
   ▼
10. JPA 把 ResultSet 映射為 List<SysUser>
   ▼
11. Controller 用 stream 把 SysUser 映射為 UserVO（過濾掉 password 字段）
   ▼
12. Spring MVC 用 Jackson 把 List<UserVO> 序列化為 JSON
   ▼
13. 響應返回瀏覽器 → JS 動態渲染表格
```

每一個箭頭背後都是一個你需要理解的「為什麼能這樣」。

## 七、安全邊界（這個 Demo 守住了什麼，沒守住什麼）

**守住了：**
- 密碼用 BCrypt 加鹽哈希存儲，數據庫被拖庫也無法反推明文
- 所有 `/login` 以外的請求默認需要認證
- HTTP-only Cookie，JS 無法讀取 JSESSIONID（CSRF 默認開啟）
- Remember-me Token 有簽名密鑰防偽造

**沒守住（為了學習簡化）：**
- 數據庫密碼是明文寫在 application.yml 裏的（生產要用環境變量 / 配置中心）
- 沒有 HTTPS（生產必須）
- 沒有登錄失敗次數限制 / 驗證碼（生產要防暴力破解）
- 沒有權限分級（所有登錄用戶權限相同，僅有 ROLE_USER）

## 八、上手後的學習路徑建議

完成本 Demo 後，建議按以下順序加深：

1. **理解 Spring Security 過濾器鏈**（這是真正在工作的人）
2. **改造為前後端分離**（去掉 Thymeleaf，前端換成 Vue/React，後端改成 REST + JSON）
3. **替換為 JWT** 認證（無狀態、不依賴 Session）
4. **加入 Redis** 存儲 Session（解決多實例共享 Session 的問題）
5. **加入角色和權限**（`@PreAuthorize`、方法級權限）

但**先把這個 Demo 親手敲一遍** — 路是一步一步走的。
