● 好，先不動代碼，講透這個類本身。

  一句話定位

  SecurityConfig 是一個「配方」類，本身不在 HTTP 請求主鏈上 —— 
  它只在應用啟動時跑一次，產出 Spring Security 真正幹活的對象。

  對照 01 文檔那張架構圖：

  HTTP 請求
     │
     ▼
  Controller → Service → Repository → Entity   ← 這是請求主鏈
     ▲
     │
  SecurityConfig（旁邊的橫切層）

  它和 Controller / Service 這些運行時組件不同，它是配置時組件。一旦應用啟動完，
  它就「退場」了，幹活的是它造出來的 Bean。

  ---
  它在做的事，本質上是這句話

  ▎ Spring Security 自帶一套默認行為，SecurityConfig 
  ▎ 是你告訴它「我要哪些不用默認、改成什麼樣」的地方。

  舉例：
  - 不寫 SecurityConfig：Spring Security 仍會生效 —— 所有 URL
  都需要登錄、自動生成一個醜醜的登錄頁、用戶名是 user、密碼啟動時打印在控制檯
  - 寫了 SecurityConfig：你說「/login
  公開、登錄頁用我的、用戶從數據庫查、登錄成功跳 dashboard、支持記住我 7 天」

  它本質是一張配置表，不是業務邏輯。

  ---
  它產出兩個 Bean，各司其職

  Bean 1：PasswordEncoder（共享工具）

  @Bean
  public PasswordEncoder passwordEncoder() {
      return new BCryptPasswordEncoder();
  }

  - 誰用它？ 兩個地方：
    a. DataInitializer 造測試數據時，調 encode("123456") 把明文加密
    b. Spring Security 內部驗密碼時，調 matches(用戶輸入的明文, 數據庫的密文)
  - 為什麼必須是 Bean？ 因為要保證全應用同一個編碼器 —— 加密用
  BCrypt，驗證也得用 BCrypt，否則永遠對不上
  - 為什麼不能 new 出來用？ 也能跑，但散落在各處，將來想換成 Argon2/SCrypt 要改
  N 個地方；統一成 Bean 後改一處即可

  Bean 2：SecurityFilterChain（真正幹活的東西）

  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
  ... }

  這個 Bean 才是 Spring Security
  在請求到來時實際運行的東西。它是一條過濾器鏈，每個 HTTP
  請求都要先穿過它，才能到達 Controller。

  你 filterChain(...) 方法裏寫的 .authorizeHttpRequests(...)、.formLogin(...)、.
  logout(...)、.rememberMe(...)，不是業務代碼，而是「裝配指令」 —— 你在告訴
  HttpSecurity 這個建造器：

  ▎ 「給我裝一條過濾器鏈，裏面要有：URL 
  ▎ 授權過濾器、表單登錄過濾器、登出過濾器、記住我過濾器，配置如下…」

  http.build() 那一刻，Spring Security
  把你的指令翻譯成一條有先後順序的過濾器鏈（默認 15+ 個過濾器），然後注冊到
  Servlet 容器。

  ---
  一個關鍵心智模型：配置時 vs 運行時

  應用啟動階段（配置時）：
     SecurityConfig.filterChain(http) 被調用一次
                  │
                  ▼
     產出一個 SecurityFilterChain 對象
                  │
                  ▼
     注冊到 Servlet 容器


  用戶訪問階段（運行時）：
     Browser ──▶ Tomcat ──▶ SecurityFilterChain（這個！） ──▶ DispatcherServlet
  ──▶ Controller
                                │
                                └─ SecurityConfig 已經不在這裏了

  所以：你改 SecurityConfig 一定要重啟應用才生效，因為它只在啟動時跑一次。

  ---
  它「不負責」什麼（同樣重要）

  很多人以為 SecurityConfig 應該包山包海，其實它不該碰：

  ┌──────────────────────────────────────────┬───────────────────────────────┐
  │                 它不該做                 │            該誰做             │
  ├──────────────────────────────────────────┼───────────────────────────────┤
  │ 從數據庫查用戶                           │ UserDetailsServiceImpl        │
  ├──────────────────────────────────────────┼───────────────────────────────┤
  │ 比對密碼明文/密文                        │ Spring Security 內部用        │
  │                                          │ PasswordEncoder 自動做        │
  ├──────────────────────────────────────────┼───────────────────────────────┤
  │ 渲染登錄頁 HTML                          │ PageController + Thymeleaf    │
  ├──────────────────────────────────────────┼───────────────────────────────┤
  │ 創建 Session、種 Cookie                  │ Spring Security               │
  │                                          │ 內部過濾器自動做              │
  ├──────────────────────────────────────────┼───────────────────────────────┤
  │ 業務權限判斷（這個訂單是不是這個用戶的） │ Service 層                    │
  └──────────────────────────────────────────┴───────────────────────────────┘

  它只負責「告訴框架規則是什麼」，剩下的活框架自己幹。

  ---
  用一句比喻收尾

  如果把 Spring Security 比作一家保安公司：
  - SecurityConfig =
  你給保安公司寫的值班守則（哪幾道門查證、哪幾道門放行、丟鑰匙了找誰登記）
  - SecurityFilterChain = 真正在門口執勤的保安隊
  - UserDetailsService = 員工花名冊
  - PasswordEncoder = 對工牌真偽的驗證儀

  守則寫好交出去，保安公司自己會排班執勤 —— 你不用親自到門口查證。

  ---
  要繼續挖哪一塊？可選方向：
  1. HttpSecurity 那條鏈式調用為什麼這麼設計（Builder 模式 + DSL）
  2. SecurityFilterChain 內部到底有哪些過濾器、按什麼順序
  3. 為什麼 Spring Security 6 不再用 WebSecurityConfigurerAdapter（從繼承到
  Bean）
  4. @EnableWebSecurity 這個註解到底開啟了什麼
