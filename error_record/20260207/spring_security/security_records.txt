================================================================================================================================================================================================
 登录核心主干

  1. 用户提交凭证
          ↓
  2. 系统拦截，触发认证
          ↓
  3. 根据凭证，从数据库查找用户
          ↓
  4. 验证凭证是否匹配
          ↓
     ┌────┴────┐
    失败      成功
     ↓         ↓
  报错      建立身份状态（Session）
                ↓
           后续请求携带状态
                ↓
           系统识别身份 → 放行

  ---
  三句话总结：

  1. 认证 — 你是谁？（凭证 vs 数据库）
  2. 建立状态 — 记住你是谁（Session）
  3. 授权 — 你能访问这里吗？（每次请求都检查）


================================================================================================================================================================================================
认证的核心分解

  用户提交凭证（用户名 + 密码）
          ↓
  1. 提取凭证
          ↓
  2. 根据用户名，去存储中加载用户
          ↓
  3. 对比密码（输入的 vs 存储的）
          ↓
     ┌────┴────┐
    不匹配    匹配
     ↓         ↓
  认证失败   生成"认证通过"的凭证对象
                ↓
           交还给上层流程（建立状态）

  ---
  一句话总结认证的本质：

  ▎ 用你提供的信息，证明你就是存储里的那个人。

================================================================================================================================================================================================
 整个登录态生命周期

  登录成功 → 存入 Session（持久）+ 存入线程（临时）
          ↓
  后续请求 → 从 Session 恢复 → 存入线程 → 用完清空
          ↓
  登出    → 清空 Session + 清空线程 + 删除 Cookie

================================================================================================================================================================================================
后续请求不是"消除认证"，而是清空线程变量：

  后续请求：
    Session → 恢复 SecurityContext → 放入线程变量 → 请求处理
          ↓
    请求结束 → 清空线程变量（不是清空 Session）

  登出才是：
    清空 Session + 清空线程变量 + 删除 Cookie


================================================================================================================================================================================================
Filter
    → 提取凭证，造 unauthenticated token
          ↓
  AuthenticationManager（接口）
          ↓
  ProviderManager（实现类，调度员）
          ↓
  AuthenticationProvider（抽象，定义骨架）
          ↓
  DaoAuthenticationProvider（查库 + 比密码）
          ↓
  返回 authenticated token
          ↓
  存入 SecurityContext → 存入 Session


================================================================================================================================================================================================
授权：

 AuthorizationFilter
    → 拦截每次请求，触发授权检查
          ↓
  AuthorizationManager（接口）
    → 决策"能不能访问"
          ↓
  RequestMatcherDelegatingAuthorizationManager（实现类）
    → 根据 URL 匹配找到对应规则
          ↓
  具体决策者：
    AuthenticatedAuthorizationManager  → 检查是否已登录
    AuthorityAuthorizationManager      → 检查是否有指定角色

  ---
  对应项目配置

  .requestMatchers("/login").permitAll()   → 任何人可访问
  .anyRequest().authenticated()            → AuthenticatedAuthorizationManager 检查

 类的职责分工

  ┌──────────────────────────────────────────────┬──────────────────────────────┐
  │                      类                      │             职责             │
  ├──────────────────────────────────────────────┼──────────────────────────────┤
  │ AuthorizationFilter                          │ 拦截请求，触发检查           │
  ├──────────────────────────────────────────────┼──────────────────────────────┤
  │ AuthorizationManager                         │ 做出放行/拒绝决策            │
  ├──────────────────────────────────────────────┼──────────────────────────────┤
  │ RequestMatcherDelegatingAuthorizationManager │ URL 匹配，分发给正确的决策者 │
  ├──────────────────────────────────────────────┼──────────────────────────────┤
  │ AuthenticatedAuthorizationManager            │ 判断是否已登录               │
  ├──────────────────────────────────────────────┼──────────────────────────────┤
  │ AuthorityAuthorizationManager                │ 判断是否有角色权限           │
  └──────────────────────────────────────────────┴──────────────────────────────┘
