  請求進來時，按順序經過這些 Filter（簡化）：

  HTTP Request
    │
    ├─ SecurityContextPersistenceFilter   [會話] 從 Session 恢復 SecurityContext
    ├─ LogoutFilter                       [會話] 識別 /logout
    ├─ UsernamePasswordAuthenticationFilter [認證①] 識別 POST /login
    ├─ RememberMeAuthenticationFilter     [認證②] 沒 Session 時靠 Cookie 重新認證
    ├─ AnonymousAuthenticationFilter      [認證兜底] 未認證的給個匿名身份
    ├─ ExceptionTranslationFilter         [異常] 捕獲 AccessDenied / AuthenticationException
    └─ AuthorizationFilter                [授權] 最後一道：你能不能訪問這個 URL
          │
          ↓
      Controller

==============================================================================================================
 二、把 Spring Security 縮成一句話的心智模型

  ▎ 一個請求，就是一個 Authentication 物件被一群 Filter 接力處理的過程。

  - 認證類 Filter：嘗試創建 Authentication 並寫進 SecurityContext
  - 授權類 Filter：讀取 SecurityContext 判斷能不能放行
  - 會話類 Filter：負責 SecurityContext 在請求之間的持久化/恢復/清除
  - 防護類 Filter：CSRF / CORS / Headers，跟身份無關，純粹的安全增強

 三、一個能用一輩子的調試習慣

  每當你看不懂某個 Spring Security 行為時，問自己三個問題就夠了：

  這三問，幾乎能解掉 90% 的 Spring Security 玄學問題。

 分析思路拆解

  第一步：读日志，找异常信号

  你给的日志里有一行关键信息：

  Securing POST /login

  这里的 /login 是 Spring Security 内部默认的表单登录处理路径。但我知道后端 Controller 写的是 /api/auth/login（前面分析架构时已经读过代码）。

  两个路径不一致 → 请求根本没打到 Controller → 所以抛错。

  ---
  第二步：反推「谁传了错误的 URL」

  请求是前端发的，所以问题在前端。直接去看 auth.js 的 login 函数：

  const res = await api.post('/login', formData)  // ← 错了

  ---
  第三步：顺手检查请求格式

  看到 URL 问题的同时，我注意到请求体的构建方式：

  const formData = new URLSearchParams()
  formData.append('username', username)

  URLSearchParams 发出去是 application/x-www-form-urlencoded（表单格式）。

  但后端是：

  public ResponseEntity<...> login(@RequestBody LoginRequest request, ...)

  @RequestBody 期望的是 application/json。

  即使 URL 改对了，格式也是错的，登录仍然会失败。所以两个问题一起修。

  ---
  关键方法论总结

  日志给的症状（/login 被拦截）
      ↓ 对比已知事实（Controller 是 /api/auth/login）
      ↓ 定位矛盾点（URL 不匹配）
      ↓ 找源头（谁构造了这个请求 → auth.js）
      ↓ 读代码，顺带检查相关细节（请求格式）
      ↓ 一并修复

  核心是**「用已知的代码结构去解释日志里的异常」**，而不是凭感觉猜。因为前面做架构分析时已经把两端代码都读过了，所以看到日志的第一秒就能对上号。

它定义的那套状态机（NEW → INITIALIZED → STARTED → STOPPED → DESTROYED）是怎么协同整棵容器树的启动与停止的

 状态机：12 个状态，不是 4 个

  如果只看四个方法，你以为 Lifecycle 是个简单的"开/关"模型。真实情况比这复杂得多。

  Tomcat 的 Lifecycle 定义了一个完整的状态机：

         NEW（刚创建）
           │
           │ init()
           ▼
     INITIALIZING
           │
           ▼
     INITIALIZED（初始化完成，但还没启动）
           │
           │ start()
           ▼
     STARTING_PREP（启动准备）
           │
           ▼
       STARTING
           │
           ▼
       STARTED（运行中）
           │
           │ stop()
           ▼
     STOPPING_PREP
           │
           ▼
       STOPPING
           │
           ▼
       STOPPED（已停止）
           │
           │ destroy()
           ▼
     DESTROYING
           │
           ▼
     DESTROYED（已销毁，对象生命结束）

     FAILED（任何阶段失败都进入此态）


 ┌───────────────────────┬──────────────────┬───────────────────────────────────────┐
  │         事件          │     触发时机     │               典型用途                │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ BEFORE_INIT_EVENT     │ init() 开始前    │ 提前准备资源                          │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ AFTER_INIT_EVENT      │ init() 完成后    │ 注册 JMX、初始化辅助资源              │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ BEFORE_START_EVENT    │ start() 开始前   │ 启动前的最后调整                      │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ CONFIGURE_START_EVENT │ 启动配置阶段     │ 解析配置（HostConfig 在这里干活）     │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ START_EVENT           │ 正在启动         │ （较少用）                            │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ AFTER_START_EVENT     │ start() 完成后   │ 启动完成后的报告、二次部署            │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ BEFORE_STOP_EVENT     │ stop() 开始前    │ 拒绝新请求、保存状态                  │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤

  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ STOP_EVENT            │ 正在停止         │ （较少用）                            │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ AFTER_STOP_EVENT      │ stop() 完成后    │ 清理                                  │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ BEFORE_DESTROY_EVENT  │ destroy() 开始前 │ 注销 JMX、回收资源                    │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ AFTER_DESTROY_EVENT   │ destroy() 完成后 │ 最终清理                              │
  ├───────────────────────┼──────────────────┼───────────────────────────────────────┤
  │ PERIODIC_EVENT        │ 后台线程周期触发 │ 定期任务（autoDeploy 的扫描就靠这个） │
  └───────────────────────┴──────────────────┴───────────────────────────────────────┘
