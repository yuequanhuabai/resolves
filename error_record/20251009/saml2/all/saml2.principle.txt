● SAML2 骨架

  一、目标

  跨域单点登录（SSO）：用户在 A 系统登录一次，就能访问 B、C、D
  等系统，无需重复登录。

  核心问题：B 系统怎么信任"这个用户已经在 A 登录过了"？ —— SAML2
  就是解决这个跨域信任传递的协议。

  ---
  二、三个核心角色

  ┌──────────────┬────────────────┬─────────────────────────────────────────┐
  │     角色     │      是谁      │                  职责                   │
  ├──────────────┼────────────────┼─────────────────────────────────────────┤
  │ User（Princi │ 用户/浏览器    │ 发起访问，充当信息搬运工（通过浏览器重  │
  │ pal）        │                │ 定向传递消息）                          │
  ├──────────────┼────────────────┼─────────────────────────────────────────┤
  │ SP（Service  │ 业务系统（B、C │ 提供服务，但自己不做认证，依赖 IdP      │
  │ Provider）   │ 、D）          │ 的认证结果                              │
  ├──────────────┼────────────────┼─────────────────────────────────────────┤
  │ IdP（Identit │ 认证中心（A）  │ 唯一负责认证用户身份的权威方            │
  │ y Provider） │                │                                         │
  └──────────────┴────────────────┴─────────────────────────────────────────┘

  关系本质：SP 和 IdP 之间有预先建立的信任关系（通过交换证书/元数据），IdP
  说"这人是张三"，SP 就信。

  ---
  三、核心流程（SP-Initiated，最常见）

  User ——访问——> SP
    SP 发现用户未登录
  SP ——发 AuthnRequest——> IdP  （经浏览器重定向）
    IdP 让用户登录（如果还没登录）
  IdP ——发 Response(Assertion)——> SP  （经浏览器 POST 回来）
    SP 验证 Assertion 的签名，提取用户身份
  SP ——放行——> User

  ---
  四、核心对象（协议消息）

  ┌──────────┬─────────────┬───────────────────────────────────────────────┐
  │   对象   │    定位     │                     功能                      │
  ├──────────┼─────────────┼───────────────────────────────────────────────┤
  │ AuthnReq │ SP → IdP    │ "请帮我认证这个用户"                          │
  │ uest     │ 的请求      │                                               │
  ├──────────┼─────────────┼───────────────────────────────────────────────┤
  │ Response │ IdP → SP    │ 装载认证结果的信封                            │
  │          │ 的响应      │                                               │
  ├──────────┼─────────────┼───────────────────────────────────────────────┤
  │ Assertio │ Response 里 │ "我（IdP）证明：这个人是张三，认证时间是X，有 │
  │ n        │ 的核心载荷  │ 效期到Y"                                      │
  ├──────────┼─────────────┼───────────────────────────────────────────────┤
  │ Metadata │ 预配置的信  │ SP 和 IdP 各自公布自己的端点地址、证书、支持  │
  │          │ 任契约      │ 的绑定方式，这是信任关系的基础                │
  └──────────┴─────────────┴───────────────────────────────────────────────┘

  ---
  五、信任如何成立

  整个机制的安全基石就一句话：

  ▎ IdP 用私钥对 Assertion 做数字签名，SP 用预先持有的 IdP 公钥验签。

  签名验证通过 = SP 确信这个 Assertion 确实来自 IdP，且未被篡改。

  ---
  一句话总结：SAML2 = IdP 替 SP 认证用户，通过浏览器重定向传递签名的 XML
  断言来传递信任。
